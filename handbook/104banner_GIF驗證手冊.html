技術筆記 · 檔案格式與後端驗證

一張 928×322 的 banner，
為什麼每一幀都不一樣大？

本文整理 104banner_上傳版.gif 的逐幀結構分析過程，釐清「畫布尺寸」與「每幀 local image 尺寸」的差異， 並給出前後端在驗證 GIF 規格時應該分別檢查的項目，附完整 PHP 實作。

01問題背景

起初用 Pillow 檢查這支 GIF 時，回報「所有幀的 size 都是 928×322」——這個結論表面上沒有語法錯誤， 但其實毫無意義：如果每一幀真的都跟畫布一樣大且完全沒有變動範圍的資訊，動畫根本不會動。

問題根源：Pillow 的 im.size 讀到的永遠是 GIF 的「邏輯畫布尺寸」（Logical Screen Descriptor）， 不是每一幀實際寫入檔案的原始子矩形尺寸。用同一個 API 查 227 次，只是重複讀了 227 次畫布大小。

後來用 ImageMagick 的 identify 直接讀取每一幀的 Image Descriptor block， 才拿到跟同事一致的「原始 local image 尺寸＋offset」，逐行核對 227 幀 零誤差。

02畫布尺寸 vs. 每幀尺寸

GIF 為了節省檔案大小，只會把「相對上一幀有變動的區域」存成一個小矩形（local image）， 沒有變動的部分不會重複寫入。播放時，播放器依照每幀的 offset X / offset Y 把這個小矩形疊加回完整畫布， 再依 disposal method 決定是否清除殘影。

每幀 Local Size

純粹是編碼優化的副產物，同樣視覺效果用不同工具轉檔，幀數與每幀大小可能完全不同，不適合當驗證條件。

04後端該驗證什麼

後端驗證的重點要跟「每一幀 size」脫鉤——每幀 local size 只是編碼細節，不穩定也不該當作規格條件。 真正該驗證的是畫布尺寸與檔案大小。

驗證項目對應資料為什麼要驗
畫布尺寸Logical Screen Descriptor（檔頭固定 byte）確保 banner 版位符合規格
檔案大小檔案 byte 數控制 CDN 流量與載入速度
幀數／總播放時長（可選）逐幀 duration 加總避免動畫過長影響效能
每幀 local size（不建議）Image Descriptor block純編碼細節，跟畫面規格無關
重點：GIF 檔頭的寬高存在固定位置（第 6~9 byte，小端序），不需要完整解碼就能讀到， 速度快，適合上傳當下即時驗證。

05PHP：畫布尺寸驗證（getimagesize()）

PHP 內建函式，只讀檔頭 metadata，不會解碼整張圖，適合上傳時即時驗證。

validate-gif-canvas.php 
<?php
function validateGifCanvasSize(string $filePath, int $expectedWidth, int $expectedHeight, int $maxBytes): array
{
    if (!file_exists($filePath)) {
        throw new RuntimeException("檔案不存在: {$filePath}");
    }

    $info = getimagesize($filePath);
    if ($info === false || $info[2] !== IMAGETYPE_GIF) {
        throw new RuntimeException("不是有效的 GIF 檔案");
    }

    [$width, $height] = $info;
    $fileSize = filesize($filePath);

    $errors = [];
    if ($width !== $expectedWidth || $height !== $expectedHeight) {
        $errors[] = "尺寸不符,預期 {$expectedWidth}x{$expectedHeight},實際 {$width}x{$height}";
    }
    if ($fileSize > $maxBytes) {
        $errors[] = "檔案過大: {$fileSize} bytes,上限 {$maxBytes} bytes";
    }

    return [
        'valid' => empty($errors),
        'width' => $width,
        'height' => $height,
        'file_size' => $fileSize,
        'errors' => $errors,
    ];
}

// 使用範例
$result = validateGifCanvasSize(
    '/path/to/104banner.gif',
    928,
    322,
    2 * 1024 * 1024 // 2MB
);

if (!$result['valid']) {
    foreach ($result['errors'] as $err) {
        echo "驗證失敗: {$err}\n";
    }
}

06PHP：Laravel 上傳驗證

如果在 Laravel 專案裡做上傳驗證，可以直接用內建的 dimensions rule。

StoreBannerRequest.php 
$request->validate([
    'banner' => [
        'required',
        'file',
        'mimes:gif',
        'max:2048', // KB
        Rule::dimensions()->width(928)->height(322),
    ],
]);

07PHP：幀數／動畫總長度驗證（需要 Imagick）

getimagesize() 拿不到幀數跟 duration，這部分要靠 Imagick 擴充套件逐幀讀取。

validate-gif-animation.php 
<?php
function validateGifAnimation(string $filePath, int $maxTotalMs): array
{
    if (!extension_loaded('imagick')) {
        throw new RuntimeException("需要安裝 Imagick 擴充套件");
    }

    $imagick = new Imagick($filePath);
    $frameCount = $imagick->getNumberImages();

    $totalMs = 0;
    foreach ($imagick as $frame) {
        // getImageDelay() 回傳單位是 1/100 秒
        $totalMs += $frame->getImageDelay() * 10;
    }
    $imagick->clear();

    $errors = [];
    if ($totalMs > $maxTotalMs) {
        $errors[] = "動畫總長 {$totalMs}ms 超過上限 {$maxTotalMs}ms";
    }

    return [
        'valid' => empty($errors),
        'frame_count' => $frameCount,
        'total_duration_ms' => $totalMs,
        'errors' => $errors,
    ];
}
沒裝 ext-imagick 的話，先確認 php -m 有沒有列出來， 沒有就依部署環境安裝，例如 apt install php-imagick。

08驗證檢查清單

  • ✓用 getimagesize() 驗證畫布尺寸是否符合規格（928×322）
  • ✓用 filesize() 驗證檔案大小是否超過上限
  • ✓如需控管動畫節奏，用 Imagick 加總 duration 驗證總長度
  • ✗不要拿「每一幀的 local image size」當驗證條件——那是編碼優化的副產物，不代表畫面規格
  • ✓如需做視覺比對，先用 convert input.gif -coalesce frame_%03d.png 攤平成完整畫面再比對
