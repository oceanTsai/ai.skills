控制反轉（IoC）與
依賴注入（DI）

這份手冊延續 SOLID 課程與 FP 手冊的編排方式，用你熟悉的前端與 Spring Boot 背景作為橋樑，帶你理解 IoC / DI 這對經常被一起提及、卻常被混為一談的概念。全文以 TypeScript 為主要示範語言。

前言：為什麼要學這個

如果你寫過 Spring Boot，一定寫過 @Autowired；如果你寫過 NestJS，一定寫過 @Injectable()。這兩個裝飾器背後，其實是同一套設計思想：控制反轉（Inversion of Control, IoC），而依賴注入（Dependency Injection, DI）是實現它最常見的手法。

這份手冊不是要你背誦定義，而是要讓你在讀完之後能自己回答一個問題：「這段程式碼的『控制權』，原本在誰手上，現在被反轉到誰手上了？」能穩定回答這個問題，你就真正理解 IoC 了。

你將會學到 IoC 的本質、DI 作為 IoC 的一種實現方式、三種注入手法、一個手寫的迷你 DI Container、以及 DI 與 SOLID 中依賴反轉原則（DIP）之間常被搞混的關係。

Ch1．耦合的痛 — 沒有 DI 時的世界

先看一段「看起來沒問題」的程式碼：

class MySQLUserRepository {
  findById(id: string) {
    // 直接連線查詢資料庫
    return { id, name: "Ocean" };
  }
}

class UserService {
  private repo = new MySQLUserRepository(); // 問題在這裡

  getUser(id: string) {
    return this.repo.findById(id);
  }
}

const service = new UserService();

問題出在 UserService 內部用 new MySQLUserRepository() 直接建立了它的依賴。這代表：

  • 要幫 UserService 寫單元測試，就一定會連到真的資料庫（或者你得改動它的原始碼才能換成假的）。
  • 將來想從 MySQL 換成 MongoDB，得回去改 UserService 的內部程式碼。
  • UserService 同時扛了「怎麼取得資料」跟「怎麼組裝自己」兩個責任。
前端視角的類比 這就像一個 React 元件在內部寫死 fetch("https://prod-api.example.com")，而不是透過 props 或 context 把 API client 傳進來。畫面沒壞，但你永遠沒辦法在測試環境或 Storybook 裡替換掉它。

這種「自己主動去找/建立依賴」的寫法，責任在使用者自己身上——這正是下一章要反轉的「控制權」。

Ch2．控制反轉（IoC）是什麼

IoC 是一個原則，不是一個具體的技術。它的主張很簡單：

IoC 的核心主張 把「決定要用哪個實作、何時建立它」的控制權，從物件自己手上，轉移到外部（一個框架、容器，或呼叫者）。

注意，IoC 反轉的「控制」不只限於依賴的建立，還包含更廣義的「流程控制」。你熟悉的例子其實到處都是：

情境傳統做法（自己控制）IoC 之後（框架控制）
UI 事件 你的程式主動 while 迴圈輪詢滑鼠狀態 瀏覽器決定何時呼叫你的 onClick
Spring Bean new UserService() 自己 new 依賴 Spring Container 決定何時建立、注入什麼實作
NestJS Module Service 內部 import 並 new Repository Nest IoC Container 依 Module 設定注入
React Hook 元件自己管理生命週期時序 React 決定何時呼叫 useEffect

所以「事件驅動」「模板方法模式」「框架的生命週期 hook」都是 IoC 的表現形式。依賴注入（DI）只是 IoC 家族裡，專門處理「依賴要怎麼給到物件手上」這個子問題的其中一種手法。其他手法還包括 Service Locator、事件/觀察者模式等，但 DI 是目前業界最主流、最容易測試的一種。

· · ·

一句話釐清常見混淆

IoC 是目標（誰該掌握控制權），DI 是手段（怎麼把依賴送過去）。就像「解耦」是目標、「介面」是達成解耦的手段之一，兩者是不同層次的詞，不是同義詞。

Ch3．依賴注入（DI）是什麼

把 Ch1 的例子用 DI 改寫：

interface UserRepository {
  findById(id: string): { id: string; name: string };
}

class MySQLUserRepository implements UserRepository {
  findById(id: string) {
    return { id, name: "Ocean" };
  }
}

class UserService {
  // 依賴透過建構子「被傳進來」，而不是自己 new
  constructor(private repo: UserRepository) {}

  getUser(id: string) {
    return this.repo.findById(id);
  }
}

// 組裝的責任被移到外部
const repo = new MySQLUserRepository();
const service = new UserService(repo);

差異看起來很小，但意義重大：UserService 現在只依賴介面 UserRepository，完全不知道也不在乎背後是 MySQL、還是測試用的假物件。「組裝」這件事被抽到外部，由呼叫者（或之後會提到的容器）決定。

寫單元測試時，你可以自由替換掉真正的實作：

class FakeUserRepository implements UserRepository {
  findById(id: string) {
    return { id, name: "測試用假資料" };
  }
}

const service = new UserService(new FakeUserRepository());
// 完全不用連資料庫，測試秒級完成
DI 的正式定義 一個類別不自行建立它所依賴的物件，而是由外部將依賴「注入」進來。類別只透過抽象（介面/型別）與依賴互動，不知道具體實作是誰。

Ch4．三種注入方式

DI 的「注入」實際上有三種常見手法，各有取捨：

1. 建構子注入（Constructor Injection）— 最常用

class UserService {
  constructor(private repo: UserRepository) {}
}

依賴在物件建立的那一刻就必須給齊，物件不會處於「半初始化」的狀態。這是 Spring、NestJS、Angular 的預設慣例，也是你應該優先選擇的方式。

2. Setter 注入（Property/Setter Injection）

class UserService {
  private repo!: UserRepository;
  setRepo(repo: UserRepository) {
    this.repo = repo;
  }
}

適合「依賴是選填的」或「依賴要在建立後動態替換」的情境，但代價是物件建立後、setRepo 呼叫前，是一個不完整的狀態，容易造成「忘記設定就用」的執行期錯誤。

3. 介面注入（Interface Injection）

interface RepositoryAware {
  setRepository(repo: UserRepository): void;
}

class UserService implements RepositoryAware {
  private repo!: UserRepository;
  setRepository(repo: UserRepository) {
    this.repo = repo;
  }
}

強制要求「需要某依賴的類別」實作對應的注入介面，由容器統一呼叫該方法完成注入。這種方式現在較少見（比較常在較舊的 Java 框架，如早期的 Avalon、PicoContainer 中看到），因為它會讓類別多背負一個跟業務邏輯無關的介面。

實務建議 預設一律用建構子注入。只有在依賴是「可選、可能不存在」或者需要在物件生命週期中動態切換時，才考慮 Setter 注入。介面注入在現代 TypeScript/JavaScript 生態中幾乎不會用到，知道它存在即可。

Ch5．手寫一個迷你 DI 容器

讀懂 DI 最快的方式，是自己動手寫一個簡化版的容器（IoC Container）。它要解決的核心問題是：「當有人要拿 A，而 A 依賴 B、B 又依賴 C 時，容器要能自動把整條依賴鏈組裝起來。」 

// 一個極簡的 DI Container：用 Map 儲存「型別」到「建立方式」的對應
type Factory<T> = (container: MiniContainer) => T;

class MiniContainer {
  private factories = new Map<string, Factory<any>>();
  private singletons = new Map<string, any>();

  // 註冊一個依賴的建立方式
  register<T>(token: string, factory: Factory<T>, singleton = true) {
    this.factories.set(token, factory);
    if (singleton) this.singletons.set(token, undefined);
  }

  // 解析（resolve）出實例，自動處理依賴鏈
  resolve<T>(token: string): T {
    if (this.singletons.has(token) && this.singletons.get(token) !== undefined) {
      return this.singletons.get(token);
    }
    const factory = this.factories.get(token);
    if (!factory) throw new Error(`Token "${token}" 尚未註冊`);

    const instance = factory(this); // 把 container 傳入，讓 factory 自己 resolve 它的依賴
    if (this.singletons.has(token)) {
      this.singletons.set(token, instance);
    }
    return instance;
  }
}

接著用它組裝 Ch3 的例子：

const container = new MiniContainer();

container.register<UserRepository>("UserRepository", () => new MySQLUserRepository());

container.register<UserService>("UserService", (c) =>
  new UserService(c.resolve<UserRepository>("UserRepository"))
);

const service = container.resolve<UserService>("UserService");

這就是 Spring Container、NestJS IoC Container 在做的事情的核心骨架：維護一張「token → 建立方式」的對應表，並在 resolve 時遞迴地把整條依賴鏈組裝出來。真正的框架還會加上：

  • 生命週期管理：singleton / transient / request-scoped。
  • 裝飾器 + Metadata Reflection：自動從 @Injectable() 的建構子參數型別推斷要注入什麼，不用手動 register。
  • 循環依賴偵測：A 依賴 B、B 又依賴 A 時要能報錯而不是無限遞迴。
類比：GAS 的觀點 你在 B Team 自動化系統中用過 Strategy Pattern 做通知策略切換，其實思路是相通的——NotifyLib 內部「決定要用哪個通知策略」的邏輯，也可以進一步抽成「外部注入策略物件」，而不是在類別內部 if/else 判斷該用哪個策略。這正是 DI 在解決的問題的一種延伸。

Ch6．從 Spring 到 NestJS — 裝飾器背後的真相

你寫過的 Spring 與 NestJS 程式碼，其實對應著幾乎一樣的心智模型：

概念Spring Boot（Java）NestJS（TypeScript）
標記「這是可被注入的服務」 @Service / @Component @Injectable()
宣告要注入什麼依賴 @Autowired（或建構子注入） 建構子參數 + TypeScript 型別
分組管理 Bean/Provider @Configuration @Module()
容器本體 ApplicationContext Nest IoC Container 
// NestJS 範例
@Injectable()
class MySQLUserRepository implements UserRepository {
  findById(id: string) { /* ... */ }
}

@Injectable()
class UserService {
  // Nest 容器在啟動時會「讀取這個建構子參數的型別」，
  // 自動去容器裡找對應的 Provider 注入進來——
  // 這就是 Ch5 那個 resolve() 遞迴過程的自動化版本
  constructor(private repo: MySQLUserRepository) {}
}

@Module({
  providers: [MySQLUserRepository, UserService],
})
class AppModule {}

關鍵差異在於：Nest 能透過 reflect-metadata 在編譯期把 TypeScript 的型別資訊保留到執行期，所以它「看得到」建構子參數的型別，自動幫你 resolve，不需要像 Ch5 的迷你容器一樣手動註冊每一條 token。這也是為什麼 NestJS 的 tsconfig.json 必須開啟 emitDecoratorMetadata。

Ch7．DI 與 SOLID 的關係——一個常被搞混的地方

你上過 SOLID 課程，應該對依賴反轉原則（Dependency Inversion Principle, DIP）有印象。這裡要特別釐清一個常見誤解：

常見誤解 「DIP 跟 DI 是同一件事，名字幾乎一樣。」——這是錯的。

DIP 是一個設計原則，主張：

  1. 高層模組不應該依賴低層模組，兩者都應該依賴抽象。
  2. 抽象不應該依賴細節，細節應該依賴抽象。

回頭看 Ch3 的例子：UserService（高層）依賴 UserRepository 介面（抽象），而不是直接依賴 MySQLUserRepository（低層細節）——這件事本身就是 DIP。

而 DI（依賴注入）是「怎麼把這個抽象的具體實作，實際送到高層模組手上」的技術手段。你完全可以做到 DIP（依賴介面而非實作），卻不用 DI（比如用 Service Locator，或工廠函式手動傳遞）；反過來，你也可能用了 DI 框架，卻沒有做到 DIP（例如建構子參數直接寫具體類別 MySQLUserRepository，而不是介面）。

DIPDI
層次設計原則（要依賴什麼）實現技術（怎麼給依賴）
回答的問題「這個類別應該依賴誰？」「這個依賴要怎麼送進來？」
沒有它會怎樣高層邏輯被低層細節綁死依賴仍需手動 new/組裝

實務上兩者幾乎總是一起出現：先用 DIP 決定「依賴介面而非實作」，再用 DI 把介面的具體實作注入進去。這也是為什麼兩個詞常被混用——但理解它們是「原則」與「手段」的關係，能幫你在做架構決策時想得更清楚。

Ch8．常見誤區與最佳實踐

誤區一：把 DI 容器當成萬能的全域變數包

如果到處都用 container.resolve("XXX") 隨手取東西（這其實是 Service Locator 模式，不是 DI），類別看起來輕巧，實際上偷偷依賴了整個容器，反而失去了「依賴清單一看建構子就懂」的優點。優先用建構子注入，讓依賴顯性化。

誤區二：注入介面卻讓介面長得跟實作一模一樣

如果你的 interface UserRepository 完全照抄 MySQLUserRepository 的每個方法簽章，介面就只是形式上的包裝，沒有真正解耦（例如換成 MongoDB 版本時，介面可能需要完全不同的查詢方式）。介面應該站在「使用者需要什麼能力」的角度設計，而不是「現有實作有什麼方法」。

誤區三：過度注入，任何東西都做成 Provider

不是所有依賴都需要走 DI。像 Date、Math 這類穩定、無副作用、幾乎不會替換的工具，直接使用即可；只有「會變化、需要替換、需要在測試中 mock」的依賴，才值得注入。過度使用 DI 會讓程式碼淹沒在一堆 @Injectable() 與 token 字串裡，反而降低可讀性。

誤區四：循環依賴

A 的建構子需要 B，B 的建構子又需要 A，這在建構子注入下是無法解的（誰先造？）。這通常是設計上的警訊，代表 A 跟 B 職責劃分不清，該考慮拆出第三個模組，或改用 Setter 注入延後其中一方的依賴設定。

最佳實踐總結 預設用建構子注入 → 依賴永遠指向介面/抽象 → 介面依「使用者需求」設計而非「照抄實作」→ 只對「真的會變化」的依賴做 DI → 出現循環依賴時，先檢討職責劃分，而不是急著換注入方式解套。

自我測驗

Q1. 「IoC」跟「DI」的關係，下列哪個說法最準確？

A. 兩者是同義詞，可以互換使用
B. DI 是 IoC 的一種具體實現手法，IoC 是更廣義的設計原則
C. IoC 是 DI 的其中一個步驟
D. DI 只存在於 Java，IoC 只存在於前端框架

看解答 答案是 B。IoC 是「把控制權交出去」這個廣義原則，DI 是其中一種常見的實現手法（其他還有 Service Locator、事件驅動等）。
Q2. 下列哪一段程式碼沒有做到依賴注入？

A. constructor(private repo: UserRepository) {}
B. this.repo = new MySQLUserRepository();（寫在建構子內部）
C. setRepo(repo: UserRepository) { this.repo = repo; }
D. 由 NestJS 容器自動注入的建構子參數

看解答 答案是 B。只要依賴是類別「自己在內部 new 出來」，無論寫在哪個方法裡，都不是依賴注入——關鍵是依賴的建立責任有沒有被移到外部。
Q3. DIP（依賴反轉原則）跟 DI（依賴注入）為什麼常被搞混，但其實不是同一件事？
看解答 因為名字非常相似，且實務上幾乎總是一起出現。但 DIP 是「設計原則」，回答「應該依賴誰（抽象 vs 細節）」；DI 是「實現技術」，回答「依賴要怎麼被送進來」。你可以只做到其中一個，而沒做到另一個。
Q4. 為什麼「到處呼叫 container.resolve() 拿依賴」不算是好的 DI 實踐？
看解答 因為那其實是 Service Locator 模式：類別表面上「沒有 new 依賴」，但實際上偷偷依賴了整個容器物件，建構子也看不出它真正需要哪些依賴，反而喪失了 DI 想要的「依賴顯性化、易於測試替換」的好處。
