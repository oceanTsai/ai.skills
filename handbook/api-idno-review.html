API Design Review · Candidate Broken-Promises

idNo 該放哪裡?
path、body,還是根本不該外露

公司規定 合理 —— 但理由是「避免 PII 進日誌」,不是「更安全防駭」。
真正的安全靠後端授權檢查;單靠 path → body,只解掉三個問題裡的一個。

Framing

先把「安全」拆成三件事

不拆開來看,就會把「日誌乾淨」誤當成「防得住攻擊」。path → body 只動得了其中一項。

問題path → body 有解?說明
PII 曝光到日誌 ✓ 有解 path / query 會被 access log、proxy、CDN、瀏覽器歷史、Referer 記錄;body 通常不會這是資料治理 / 個資合規面,不是防駭
傳輸被竊聽 — 無差 HTTPS 已保護,path 或 body 都一樣受 TLS 保護
未授權存取 (IDOR) ✗ 沒解 有權呼叫 API 的人,把別人 idNo 塞進 body 一樣查得到Broken Object Level Authorization

Your Two Instincts

你的兩個直覺都對
① 破壞了 RESTful 冪等性 — 對

查詢型 API 用 GET 才符合冪等、可 cache、語意清楚。但用 POST /search 是業界普遍接受的務實破例(GitHub、Elastic、Google 都這樣),為了敏感資料犧牲一點 REST 語意是合理取捨。純粹的兩全其美不存在,只能選「哪個代價可接受」。

② 放 body 我還是看得到 — 完全正確,這才是核心

body 擋的是「中介系統的被動記錄」(log、cache、Referer),擋不了「有權呼叫 API 的人」。對能發請求的人來說,path、body、甚至加密欄位他都看得到明文。

Field Encryption?

「把欄位加密」能解決嗎?幾乎沒用

要看加密誰、防誰:

傳輸加密 HTTPS 已經做了,再加一層對防呼叫者沒意義
欄位級加密到
連呼叫端都拿不到明文		 但前端要能查詢就得先持有明文 idNo,加密就失去意義了

除非改成「呼叫端根本不持有 idNo」—— 這就導向真正的解法。

The Real Fix

opaque token + 後端授權

別讓 idNo 當對外查詢鍵。流程改成:

  1. 呼叫端手上只有系統發的 opaque reference(如 candidateRef,一段 hash 或加密後、無法反推、無法枚舉的字串)
  2. API 收到 ref → 後端內部映射回真正的 idNo
  3. 做物件層級授權檢查 —— 這家公司有沒有權看這個人
  4. 通過後才查詢
URL / body 都沒有明文身分證字號 → 日誌乾淨
呼叫端就算看得到 ref 也沒用 → 反推不出 idNo、也猜不到別人的 ref
配合後端授權檢查 → 擋掉 IDOR

Recommendation

給團隊的建議
短期 · Ship it

照公司規定用 POST /search 沒問題,至少 log 乾淨了。命名 applicable-jobs/search 也 OK。

中長期 · Invest

把 idNo 換成內部 reference,加上後端的物件層級授權。那個不做,搬 body 或加密欄位都只是心理安慰。

One-line conclusion

規定合理,但理由是「避免 PII 進日誌」;真正的安全靠後端授權檢查,單靠 path → body 只解三分之一。
