API 設計模式與最佳實踐

給前端工程師的後端 API 設計觀念手冊 — 從「怎麼串」到「為什麼要這樣設計」

12 章 · 自學導向 · 範例以 REST + JS/TS/fetch 為主 · Traditional Chinese
CHAPTER 01

為何前端工程師也要懂 API 設計？

Why frontend engineers should care about API design

前端工程師每天都在「消費」API，卻很少被要求去「設計」API。但實務上，前端往往是第一個發現 API 設計問題的人：欄位命名不一致、錯誤格式每個 endpoint 都不同、分頁邏輯每個工程師寫法都不一樣。這些問題不會出現在後端的單元測試裡，卻會直接變成前端的維護負擔。

比喻 API 設計就像餐廳的菜單設計。菜單設計得好，顧客（前端）點餐清楚、不會誤會；設計得差，顧客得反覆確認「這個套餐到底有沒有附湯」，廚房（後端）也得不斷被打斷去解釋菜單。API 文件不清楚時，前端工程師花的時間往往不是在「寫程式」，而是在「猜後端到底想表達什麼」。

API 是前後端之間的合約

CONCEPT · Contract

把 API 想成前端與後端之間簽的一份「合約」：後端承諾「你打這個 endpoint、帶這些參數，我保證回傳這個結構」；前端則依賴這份承諾去寫程式。合約一旦不穩定（欄位今天叫 userName、明天變成 user_name），前端程式碼就會不斷崩壞。

學 API 設計，本質上不是要前端去寫後端程式，而是要學會「看懂並批判這份合約」——知道什麼是好的設計、什麼是地雷，才能在 code review、跨team溝通時提出有意義的意見，而不是照單全收。

前端最常撞到的四種 API 痛點

痛點典型症狀根本原因
命名不一致同一個「使用者ID」，有的叫 id、有的叫 userId、有的叫 user_id缺乏團隊層級的命名規範
狀態碼亂用找不到資源回 200 但 body 塞 {error: "not found"}把 HTTP 語意當裝飾，不當合約
錯誤格式每次不同有時候是字串、有時候是物件、有時候欄位叫 msg 有時叫 message沒有統一的錯誤設計規範
分頁邏輯各自為政有的 API 用 page/size，有的用 offset/limit，有的用 cursor缺乏團隊層級的 API 設計指南（這正是這份手冊要解決的）

本章結論

  • API 是前後端的合約，前端有資格、也有必要理解它的設計原則
  • 大部分「串接痛苦」不是技術問題，是設計一致性問題
  • 接下來的章節會從 REST 的核心思維開始，逐步建立一套可以拿來檢核任何 API 的心智模型
CHAPTER 02

資源導向思維：REST 到底在講什麼

REST is a resource-oriented mindset, not just "using HTTP"

很多人以為「用 HTTP + JSON」就叫 RESTful，但 REST（Representational State Transfer）真正的核心是一種建模方式：把系統看成一群「資源（Resource）」，每個資源有自己的網址（URI），透過統一的一組動詞（HTTP Method）去操作它，而不是把每個「動作」都寫成一個專屬的 endpoint。

心智模型 URL 是名詞，HTTP Method 是動詞。看到 URL 裡出現動詞（/getUser、/deleteOrder），幾乎都是 REST 設計上的警訊——那其實是 RPC（Remote Procedure Call）風格，只是套了 HTTP 的殼。

RPC 風格 vs REST 風格

✕ RPC 風格（動詞塞進 URL）
GET  /api/getUserList
POST /api/createUser
POST /api/deleteUserById?id=42
POST /api/updateUserEmail
✓ REST 風格（資源為名詞，動詞交給 HTTP Method）
GET    /api/users          // 取得使用者列表
POST   /api/users          // 建立使用者
DELETE /api/users/42       // 刪除 id=42 的使用者
PATCH  /api/users/42       // 更新 id=42 的使用者（部分欄位）

RPC 風格不是「錯」，在某些場景（例如觸發一個沒有明確對應資源的動作，像「重新計算報表」）甚至更直覺。但問題在於：如果一個系統裡 REST 和 RPC 風格混雜、沒有規則，前端就無法「猜」出 API 的樣子，每個 endpoint 都要重新查文件。

資源的巢狀關係

CONCEPT · Nested Resources

資源之間常有「從屬關係」，例如「某個使用者的訂單」可以設計成 /users/42/orders，語意上代表「屬於 user 42 的 orders 集合」。

但巢狀不宜太深。/users/42/orders/7/items/3/reviews 這種四層巢狀，會讓 URL 難以閱讀、難以在前端做路由管理。實務上的經驗法則是巢狀最多兩層，更深的關係改用查詢參數或獨立 endpoint 表達，例如 /reviews/9?orderId=7 或直接 /order-items/3/reviews。

為什麼這樣設計 REST 的統一介面（GET/POST/PUT/PATCH/DELETE 這固定幾個動詞）讓前端只需要學「一套規則」就能操作「所有資源」，不需要為每個資源記憶一套專屬的 API 命名方式。這正是 REST 相比純 RPC 的最大優勢：可預測性（predictability）。

本章結論

  • REST 的本質是「資源導向」：URL 代表名詞、HTTP Method 代表動詞
  • URL 裡出現動詞通常是設計警訊，代表退化成 RPC 風格
  • 巢狀資源方便表達從屬關係，但深度建議控制在兩層以內
CHAPTER 03

HTTP Method 的正確語意

GET / POST / PUT / PATCH / DELETE — safe? idempotent?

每個 HTTP Method 都帶有兩個關鍵性質：安全性（Safe）——呼叫它是否會改變伺服器狀態，以及冪等性（Idempotent）——呼叫一次和呼叫十次，結果是否相同。這兩個性質不是規範上的裝飾，瀏覽器、CDN、Proxy、瀏覽器的自動重試機制都依賴這些性質做決策。

Method用途Safe（無副作用）Idempotent（冪等）
GET讀取資源✓ 是✓ 是
POST建立資源 / 觸發動作✕ 否✕ 否
PUT整體覆蓋資源✕ 否✓ 是
PATCH部分更新資源✕ 否視實作而定，理想上是
DELETE刪除資源✕ 否✓ 是
常見誤用 用 GET 去觸發有副作用的動作（例如 GET /api/users/42/send-reset-email），是最危險的誤用之一。瀏覽器的預先載入（prefetch）、爬蟲、CDN 快取都可能「不小心」呼叫到 GET，如果背後有副作用，就會變成無法預期的災難——曾有真實案例是爬蟲把「刪除」的 GET 連結全部點過一輪，資料庫瞬間清空。

PUT 與 PATCH：整體覆蓋 vs 部分更新

PUT — 語意上是「整份取代」
PUT /api/users/42
{
  "name": "Ocean",
  "email": "ocean@example.com",
  "role": "lead"
}
// 沒有帶到的欄位，理論上會被視為要清空或還原成預設值
PATCH — 語意上是「只改我帶的欄位」
PATCH /api/users/42
{ "role": "lead" }
// 只更新 role，其他欄位維持原值
為什麼前端要在意這個 如果前端表單只想更新一個欄位（例如切換使用者角色），卻呼叫了語意是 PUT 的 endpoint 而只送一個欄位，很可能會把其他欄位「清空」——這不是前端的 bug，是雙方對 HTTP Method 語意沒有共識造成的資料損毀。串接新 API 前，先確認後端的 PUT 是真的整體覆蓋，還是其實實作成 PATCH 的行為。

本章結論

  • 每個 HTTP Method 都有明確語意：Safe（有無副作用）與 Idempotent（是否可重複呼叫）
  • GET 絕對不能有副作用——這是瀏覽器與快取機制信任的基本假設
  • PUT 是整體覆蓋，PATCH 是部分更新，兩者混用會造成資料被意外清空
CHAPTER 04

HTTP Status Code 全解析

狀態碼是合約的一部分，不是裝飾

狀態碼的作用是讓前端不需要解析 body 內容就能判斷這次請求的大方向結果。如果所有情況都回 200，再靠 body 裡的欄位判斷成功或失敗，就等於放棄了 HTTP 協定本來就提供的溝通機制，逼前端每個 API 都要寫一套客製化的「解析成功/失敗」邏輯。

反模式 · "永遠 200" HTTP 200 + { "success": false, "error": "user not found" } 這種設計，讓瀏覽器的 devtools Network 面板、Sentry 之類的錯誤監控、API Gateway 的告警規則全部失效——它們都是依賴狀態碼在判斷「這次請求有沒有出錯」。

分類總覽

區間意義常用 code
2xx成功200 OK · 201 Created · 204 No Content
3xx重新導向301 永久搬移 · 304 Not Modified（快取仍有效）
4xx客戶端錯誤（前端該修的）400 · 401 · 403 · 404 · 409 · 422 · 429
5xx伺服端錯誤（後端該修的）500 · 502 · 503

前端最容易混淆的四組

401 UNAUTHORIZED「你沒登入 / token 失效」→ 前端該導去登入頁
403 FORBIDDEN「你登入了，但沒權限做這件事」→ 前端該顯示無權限畫面，不是導去登入頁
200 OK「請求成功，body 裡有內容」→ 適用於 GET、成功的 PATCH/PUT
201 CREATED「成功建立資源」→ 適用於成功的 POST，理想上會在 header 帶 Location 指向新資源網址
204 NO CONTENT「成功，但沒有內容要回」→ 適用於成功的 DELETE，前端不該嘗試解析 body
304 NOT MODIFIED「資源沒變，用你手上的快取即可」→ 常搭配 ETag／If-None-Match
400 BAD REQUEST「你送來的格式本身就有問題」（缺欄位、型別錯誤）
422 UNPROCESSABLE ENTITY「格式沒問題，但語意上不合法」（例如 email 格式對，但這個 email 已被註冊）

429 與重試機制

CONCEPT · Rate Limiting Response

429 Too Many Requests 代表你打太頻繁被限流了。好的 API 設計會在 response header 附上 Retry-After（秒數）或 X-RateLimit-Reset，讓前端知道該等多久再重試，而不是用固定的 setTimeout 亂猜。

本章結論

  • 狀態碼是給「機器」判斷用的，不是給人看的裝飾——不要用 200 包裝所有結果
  • 401 =「沒登入」，403 =「登入了但沒權限」，前端的處理方式應該完全不同
  • 429 應搭配 Retry-After，讓重試邏輯有依據，而不是憑感覺
CHAPTER 05

URL / Endpoint 設計慣例

好的 URL 讓人不用查文件就能猜對一半

CONCEPT · Predictable Naming

一套好的 URL 設計規則，應該讓工程師看過三、五個 endpoint 之後，就能「猜」出其他 endpoint 長什麼樣子。這種可預測性省下的溝通成本，遠比多花十分鐘討論命名規則划算。

常見慣例清單

  • 名詞用複數：/users 而不是 /user，集合與單一資源共用同一個 base path（GET /users 取列表，GET /users/42 取單一筆）
  • 全小寫、連字符分隔：/order-items 而不是 /orderItems 或 /Order_Items（URL 對大小寫敏感，混用容易踩雷）
  • 不要把動詞放進路徑：/users/42/activate 這類「動作型」endpoint 是可接受的例外，但應該謹慎使用，優先考慮能不能建模成資源狀態的更新（PATCH /users/42 { "status": "active" }）
  • 版本放在最前面：/v1/users，讓版本切換一目了然（詳見第 09 章）
  • 查詢參數用於「篩選」，不是用於「識別」：識別特定資源要放在路徑上（/users/42），不要用 /users?id=42
✕ 常見的命名反例
GET /User/GetById?ID=42
GET /orderItem/list
POST /users/42/UPDATE_PROFILE
GET /users?userId=42&action=fetchOne
✓ 對應的慣例寫法
GET /v1/users/42
GET /v1/order-items
PATCH /v1/users/42
GET /v1/users/42
比喻 URL 命名規則就像檔案系統的目錄結構。你不會在同一個專案裡一半資料夾用 camelCase、一半用 snake_case——URL 也是同樣的道理，一套系統裡不該有兩種命名風格並存。

本章結論

  • 好的 URL 命名是「可猜測的」，看幾個範例就能推敲出其他 endpoint
  • 資源用複數名詞、全小寫連字符，動詞盡量交給 HTTP Method 而不是塞進路徑
  • 識別資源用路徑（/users/42），篩選資源用查詢參數（/users?role=admin）
CHAPTER 06

Request 設計：分頁、過濾、排序

列表型 API 的三大標準配備

分頁：Offset-based vs Cursor-based

Offset-based（page / limit）Cursor-based（cursor / after）
範例GET /posts?page=2&limit=20GET /posts?after=eyJpZCI6NDJ9&limit=20
優點直覺、可以直接跳到第 N 頁資料在分頁過程中被新增/刪除也不會錯位或重複
缺點資料同時被新增/刪除時，翻頁可能重複看到或漏看資料不能直接跳頁（沒有「跳到第 50 頁」的概念）
適用場景資料變動不頻繁的後台列表、報表即時性高的 feed（動態消息、通知列表）
為什麼 Cursor-based 比較穩 Offset-based 分頁的本質是「跳過前面 N 筆」，如果使用者正在看第 2 頁時，第 1 頁被刪掉一筆資料，原本第 2 頁的第一筆就會往前移到第 1 頁，使用者翻頁時就會「漏看」一筆。Cursor 分頁記錄的是「上一筆看到的資料位置」，不受中間資料變動影響。

過濾與排序的慣例

典型的列表 API 查詢參數組合
GET /v1/orders?status=paid&createdAfter=2026-01-01&sort=-createdAt&page=1&limit=20

// status         → 過濾條件（精確匹配）
// createdAfter   → 過濾條件（範圍）
// sort=-createdAt→ 依 createdAt 遞減排序（負號慣例代表 DESC）
// page / limit   → 分頁
CONCEPT · 排序方向的慣例

業界常見的排序慣例是用「前綴符號」表示方向：sort=name 代表升冪，sort=-name 代表降冪，多欄位排序用逗號分隔：sort=-priority,createdAt。這比 sortBy=name&sortOrder=asc 的寫法更精簡，但團隊要事先約定好，否則前端會猜錯方向。

前端串接時該檢查的三件事

  • 分頁 meta 裡有沒有「總筆數」或「是否還有下一頁」（hasMore / totalCount），不然無法畫出正確的分頁 UI
  • 過濾參數如果不合法（例如 status=unknown_value），API 是回 400 還是直接忽略該條件？兩種行為前端的錯誤處理完全不同
  • 排序欄位是否有白名單限制？如果前端可以任意帶欄位排序，要注意這可能變成後端的效能或注入風險

本章結論

  • Offset 分頁直覺但在資料頻繁變動時會錯位，Cursor 分頁穩定但不能跳頁
  • 排序方向常用 - 前綴表示 DESC，團隊需要統一慣例
  • 分頁 API 一定要回傳「還有沒有下一頁」的資訊，否則前端無法畫分頁 UI
CHAPTER 07

Response 設計：統一格式與命名規範

同一個系統裡，所有 API 的「形狀」應該長得一樣

Envelope（包裝格式）vs 直接回傳

方案 A · 直接回傳陣列/物件（精簡）
GET /v1/users
[
  { "id": 1, "name": "Ocean" },
  { "id": 2, "name": "Nick" }
]
方案 B · Envelope 包裝（可帶 meta）
GET /v1/users
{
  "data": [
    { "id": 1, "name": "Ocean" },
    { "id": 2, "name": "Nick" }
  ],
  "meta": {
    "totalCount": 42,
    "page": 1,
    "hasMore": true
  }
}
為什麼多數團隊選 Envelope 方案 A 看似精簡，但一旦需要加分頁資訊、排序狀態等 metadata，就沒有地方放，只能塞進 HTTP header（前端較難處理，且不同 gateway 對 header 大小/命名有限制）。方案 B 犧牲一點精簡度，換來「未來要加欄位」的擴充空間，且格式在所有 endpoint 之間保持一致，前端可以寫一個共用的 response parser。

命名規範：camelCase vs snake_case

CONCEPT · 前後端命名對齊

JavaScript/TypeScript 生態的慣例是 camelCase，但許多後端語言（Python、Ruby）的慣例是 snake_case。如果後端直接把資料庫欄位（snake_case）原封不動吐給前端，前端就要在程式裡到處轉換命名，或忍受混用兩種風格。

務實的做法：在 API 這一層做一次轉換，讓 JSON response 的欄位統一用 camelCase，作為前後端的公約，資料庫內部要用什麼命名是後端自己的事，不該外露到 API 合約上。

日期時間、金額、列舉值的格式一致性

資料類型建議格式原因
日期時間ISO 8601（2026-07-07T10:30:00Z）跨時區明確、幾乎所有語言都有內建 parser
金額整數（分為單位）或字串，避免浮點數浮點數運算會有精度誤差，金額計算絕對不能用 float
列舉值（狀態）固定字串常數（"active"），不要用數字 magic number數字（status: 2）前端要對照表才知道意思，字串本身就有語意

本章結論

  • 統一用 Envelope 格式（data + meta），為未來擴充留空間
  • API 回傳的 JSON 命名風格要統一（建議對前端友善的 camelCase），不該外露資料庫內部命名習慣
  • 日期用 ISO 8601、金額避免浮點數、狀態用有語意的字串而非數字
CHAPTER 08

錯誤處理設計

好的錯誤格式讓前端可以「依照錯誤類型做不同 UI 反應」，不是只能顯示一句籠統訊息

反模式 前端最痛苦的錯誤處理方式，是後端只回一句人類可讀的訊息字串（"該 email 已被使用"），前端還要用字串比對或正規表達式去判斷這是「email 重複」還是「其他驗證錯誤」，一旦後端改了措辭，前端的判斷邏輯就整個失效。

統一錯誤格式：機器可讀 + 人類可讀分離

✓ 建議格式
HTTP 422 Unprocessable Entity
{
  "error": {
    "code": "EMAIL_ALREADY_EXISTS",
    "message": "這個 email 已經被註冊過了",
    "field": "email"
  }
}
CONCEPT · Error Code 是給程式看的，Message 是給人看的

code 是一個穩定不變的機器可讀識別碼，前端用它來決定「該做什麼」（例如把游標focus到 email 欄位、顯示特定的重新導向按鈕）；message 則是給使用者看的文字，後端可以隨時調整措辭、甚至做 i18n，而不會影響前端的判斷邏輯。

這就是「關注點分離」在 API 設計上的體現：邏輯判斷依賴穩定的 code，呈現內容依賴易變的 message。

前端如何依賴 error.code 分流處理

前端範例：依 error code 分流
try {
  await api.post('/v1/users', payload);
} catch (err) {
  const code = err.response?.data?.error?.code;

  switch (code) {
    case 'EMAIL_ALREADY_EXISTS':
      form.setFieldError('email', '此 email 已被使用');
      break;
    case 'RATE_LIMITED':
      toast.show('操作太頻繁，請稍後再試');
      break;
    default:
      toast.show('發生未知錯誤，請稍後再試');
  }
}

驗證錯誤（多欄位）的表達方式

表單同時有多個欄位驗證失敗
HTTP 400 Bad Request
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "輸入資料驗證失敗",
    "details": [
      { "field": "email", "code": "INVALID_FORMAT" },
      { "field": "age", "code": "OUT_OF_RANGE" }
    ]
  }
}

本章結論

  • 錯誤格式要統一，且區分「機器可讀 code」與「人類可讀 message」
  • 前端應該依賴 error.code 做邏輯判斷，不要對 message 字串做比對
  • 多欄位驗證錯誤要用陣列結構表達，讓前端能一次性標示所有錯誤欄位
CHAPTER 09

版本控制策略

API 一定會演進，問題只是「怎麼演進而不打破現有前端」

三種常見版本控制方式

方式範例優點缺點
URI Versioning/v1/users直觀、可在瀏覽器直接測試、快取友善同一資源的不同版本要維護多份路由
Header VersioningAccept: application/vnd.api+json;version=2URL 保持乾淨，理論上更「RESTful」不直覺，瀏覽器測試不方便，容易被忽略
Query Param/users?version=2實作簡單容易被快取層忽略版本差異，語意上version不像是資源的一部分
為什麼多數團隊選 URI Versioning URI Versioning 雖然「不夠優雅」，但實務上最容易溝通、除錯、寫文件——貼一個網址在 Slack 上，大家馬上知道是哪個版本。API 設計的優雅程度，永遠要讓步給團隊的實際維運成本。

什麼是 Breaking Change？

CONCEPT · Breaking vs Non-breaking

非破壞性變更（不需要升版）：新增一個回應欄位、新增一個可選的查詢參數、新增一個新的 endpoint。前端只要沒有寫死「只允許這些欄位」的嚴格解析邏輯，就不受影響。

破壞性變更（需要升版或協調）：刪除或改名一個既有欄位、改變欄位的資料型別（字串變數字）、改變某個狀態碼的意義、把原本可選的欄位變成必填。這些變更一旦上線，會讓「還沒更新」的前端直接壞掉。

前端視角：如何應對 API 升版

  • 解析 response 時養成「防禦性存取」習慣，用 ?.、預設值，而不是假設欄位一定存在
  • 對於列舉值（狀態字串），處理 switch/case 時一定要有 default 分支，避免後端新增一個新狀態值時前端整個壞掉
  • API 若標示 Deprecated（常見於 response header Deprecation: true 或文件標註），要主動排時程遷移，不要等到舊版被關閉才處理

本章結論

  • URI Versioning 雖不是最「純粹」的 REST 風格，但溝通與維運成本最低，是業界主流選擇
  • 新增欄位通常是安全的，刪除/改名/改型別欄位幾乎一定是破壞性變更
  • 前端寫防禦性解析程式碼，是應對 API 演進最實際的保險
CHAPTER 10

認證與授權：前端視角看懂 Session / JWT / OAuth2

Authentication（你是誰）vs Authorization（你能做什麼）

心智模型 Authentication 是「驗證身份」——你登入時證明「我是 Ocean」；Authorization 是「授權範圍」——系統決定「Ocean 可以看哪些資料、做哪些操作」。401 對應前者失敗，403 對應後者失敗（回顧第 04 章）。

三種主流機制對照

Session-basedJWT (Token-based)OAuth2
狀態存在哪伺服器端（session store）Token 本身（自帶簽名資訊）授權伺服器（Authorization Server）
前端要做什麼瀏覽器自動帶 cookie，前端幾乎無感手動把 token 放進 Authorization header導去第三方登入頁，處理 redirect + callback
擴展性多台伺服器需要共享 session store天然適合多台伺服器（不需要共享狀態）適合「用 Google/GitHub 登入」等第三方整合場景
登出即時性伺服器可立即刪除 session，立刻失效Token 發出後在到期前很難「強制」失效（除非額外做黑名單機制）依託 Authorization Server 的機制

JWT 常見誤解：它不是「加密」的

常見誤解 JWT（JSON Web Token）的中間那段 payload 只是 Base64 編碼，不是加密，任何人拿到 token 都能直接解碼看到裡面的內容（可以貼到 jwt.io 驗證）。JWT 的安全性來自「簽名（signature）」——確保內容沒被竄改，但完全不保密。因此絕對不要把密碼、身分證號碼等敏感資料塞進 JWT payload。

Access Token 與 Refresh Token

CONCEPT · 為何要兩個 Token

Access Token 存活時間短（例如 15 分鐘），用來打一般 API；Refresh Token 存活時間長（例如 30 天），只用來換發新的 Access Token。這樣設計的原因是：Access Token 短命，即使被竊取，攻擊者能用的時間窗口很短；Refresh Token 雖然長命，但只在「換發」這一個受限的動作上使用，且通常存放在更安全的地方（如 HttpOnly cookie，前端 JS 無法直接讀取，降低 XSS 竊取的風險）。

前端典型的 Token 刷新邏輯（攔截 401 自動重試）
api.interceptors.response.use(
  response => response,
  async (error) => {
    if (error.response?.status === 401 && !error.config._retried) {
      error.config._retried = true;
      await refreshAccessToken(); // 用 refresh token 換新的 access token
      return api(error.config);       // 重打一次原本失敗的請求
    }
    throw error;
  }
);

前端最常撞到的坑：CORS

為什麼會有 CORS 瀏覽器基於安全考量，預設禁止網頁對「不同 origin」的伺服器發請求（除非目標伺服器明確允許）。這不是後端故意為難前端，而是瀏覽器內建的同源政策（Same-Origin Policy）。後端需要在 response header 加上 Access-Control-Allow-Origin 等設定，才能允許特定網域的前端呼叫。CORS 錯誤永遠是「後端該設定」的問題，前端能做的很有限（開發環境用 proxy 繞過是常見的權宜之計）。

本章結論

  • Authentication（驗證身份）與 Authorization（授權範圍）是兩個不同層次的問題，分別對應 401 / 403
  • JWT 只是「簽名」不是「加密」，絕不能塞敏感資料
  • Access Token 短命 + Refresh Token 長命，是業界標準的風險控管設計
  • CORS 是瀏覽器的同源政策，需要後端主動開放，前端無法單方面解決
CHAPTER 11

冪等性與進階議題

Idempotency Key · Rate Limiting · Webhook · OpenAPI

為什麼 POST 需要 Idempotency Key

回顧第 03 章，POST 天生不是冪等的——呼叫兩次「建立訂單」，理論上會建立兩筆訂單。但現實中，網路不穩、使用者手滑連點兩次「送出」按鈕、前端重試邏輯自動重打，都可能造成「非預期的重複建立」。

CONCEPT · Idempotency Key

解法是前端在發送請求時，自己產生一個唯一識別碼（通常是 UUID），放進 Idempotency-Key header。後端收到後先檢查這個 key 是否已經處理過：如果是，直接回傳上次處理的結果，不會重複建立；如果不是，才真正執行建立動作並記錄這個 key。

前端使用範例
const idempotencyKey = crypto.randomUUID();

await api.post('/v1/orders', payload, {
  headers: { 'Idempotency-Key': idempotencyKey }
});

// 即使因網路問題重試同一個請求，只要 header 帶同一個 key，
// 後端保證不會建立第二筆訂單
為什麼這對「送出訂單」「扣款」這類 API 特別重要 這類操作一旦重複執行，代價是真實的金錢損失或業務錯誤，不像「重複建立一則留言」那樣可以事後刪除補救。凡是牽涉金流、庫存扣減的 POST API，都應該優先考慮加上 Idempotency Key 機制。

Rate Limiting：保護伺服器，也保護前端的重試邏輯

典型的 Rate Limit Header
X-RateLimit-Limit: 100        // 每個時間窗口允許的次數
X-RateLimit-Remaining: 42    // 目前剩餘次數
X-RateLimit-Reset: 1720000000 // 重置時間（unix timestamp）

好的 Rate Limit 設計會把這些資訊放在 header 讓前端可以「提前」得知快被限流了（例如剩餘次數低於 5 時，前端可以自動節流請求），而不是等到 429 才知道。

Webhook：反過來由後端主動通知前端／第三方系統

比喻 一般 API 呼叫像「打電話問對方」——你主動發出請求，等待對方回應。Webhook 則像「訂閱後由對方主動打給你」——你先登記一個網址（callback URL），事情發生時對方會主動發請求通知你，不用一直輪詢（polling）問「好了嗎？好了嗎？」。常見場景：金流服務的付款結果通知、CI/CD 系統的建置完成通知。

OpenAPI（Swagger）：契約先行

CONCEPT · Contract-First Development

OpenAPI 是一份用 YAML/JSON 描述 API 的規格（endpoint、參數、response 結構、錯誤碼都寫在裡面），可以自動產生互動式文件（Swagger UI）、自動產生前端的 TypeScript 型別、甚至自動產生 Mock Server。

「契約先行」的意思是：前後端在開始寫程式碼之前，先一起討論並定案這份 OpenAPI 規格。前端可以拿著規格直接用 Mock Server 開發，不需要等後端真的把 API 寫完；後端則按規格實作，最後對接時大部分的落差已經在設計階段被抓出來，而不是等到串接才發現「怎麼欄位名字不一樣」。

本章結論

  • 會產生實際業務後果（金流、庫存）的 POST API，應該用 Idempotency Key 防止重複執行
  • Rate Limit 資訊放在 response header，讓前端可以提前節流，而不是等 429 才反應
  • Webhook 是「反向」的 API 呼叫，適合事件通知場景，可以避免高頻率 polling
  • OpenAPI 契約先行能讓前後端在開發初期就對齊介面，大幅減少串接期的落差

實戰：重新設計一個爛 API

綜合應用前十一章的所有原則

假設你收到一份現有的、設計不佳的 API 規格，要負責提出改善建議。這種情境下，最有效率的做法是逐一對照前面章節建立的檢核清單。

爛 API 範例

✕ 原始設計
POST /api/getOrderList
Request: { "userId": 42, "pageNum": 1 }

Response: HTTP 200
{
  "success": true,
  "result": [
    { "OrderID": 1, "order_status": 2, "total": "199.00" }
  ],
  "msg": ""

逐項檢核（對照前面章節）

問題對應章節根本原因
用 POST /api/getOrderList 查詢資料Ch02 · Ch03RPC 風格，且查詢動作該用 GET（safe、可快取）卻用了 POST
錯誤時仍回 HTTP 200Ch04狀態碼被當裝飾，前端無法依賴 HTTP 協定判斷成敗
欄位命名混用 OrderID / order_status / userIdCh07缺乏統一的命名規範（PascalCase / snake_case / camelCase 混雜）
total 是字串 "199.00"Ch07金額型別不一致，前端無法直接做數值運算
order_status: 2 是數字Ch07Magic number，前端需要額外對照表才知道 2 代表什麼

提出改善建議時的溝通技巧

  • 不要只說「這個設計不好」，具體指出違反了哪個原則、會造成前端什麼實際的維護成本
  • 用「小步改善」取代「全部重寫」——如果 API 已經上線且有其他系統依賴，優先討論「新增 v2」而不是直接改壞現有合約
  • 把這份手冊的檢核表變成團隊共用的 API Review Checklist，讓「好壞」的判斷不是靠個人經驗，而是有共同標準

全書總結

  • API 設計的核心是「一致性」與「可預測性」——好的設計讓人看幾個範例就能猜對其他部分
  • HTTP 協定本身（Method、Status Code）已經提供了豐富的語意，善用它比自己發明一套規則更划算
  • 前端不需要會寫後端，但看懂這些設計原則，能大幅提升跨團隊溝通的品質與效率
