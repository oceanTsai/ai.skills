🎓 1. NestJS 基礎概念

NestJS 是 Node.js 後端框架，架構靈感來自 Angular。對前端工程師超友善，因為很多概念你都看過。

前端工程師快速理解 NestJS

前端 (React/Angular)

Component / Service / Module
用 Hooks 或 DI 注入依賴
檔案結構分明

NestJS 後端

Controller / Service / Module
用 @Injectable 注入依賴
類似的模組化架構

三大核心概念

@Controller
負責接收 HTTP 請求

就像 React 的 page component，負責「介面」。它的工作只是接收請求、呼叫 service、回傳結果。

chat.controller.ts
@Controller('chat')              // 處理 /chat 路徑
export class ChatController {
  constructor(
    private chatService: ChatService  // 注入 Service
  ) {}

  @Post('ask')
  async ask(@Body() body: AskDto) {
    return this.chatService.ask(body.question)
  }
}
@Injectable / Service
負責業務邏輯

就像 React 的 custom hook 或 utility module，把邏輯抽出來重用。Service 之間可以互相注入。

@Module
把相關元件群組起來

就像 npm 的 package，把相關 Controller + Service 組合在一起，形成可重用的「功能單位」。

chat.module.ts
@Module({
  controllers: [ChatController],
  providers: [ChatService],
  exports: [ChatService]      // 讓其他 Module 可以用
})
export class ChatModule {}
🔑 DI（Dependency Injection）

NestJS 最強大的特色：constructor 裡寫的依賴會自動注入，你不用手動 new。這對 DDD 超重要——可以輕鬆替換「實作」（例如本地測試用 mock，正式環境用真的 OpenAI）。

📐 2. DDD 核心概念

DDD（Domain-Driven Design，領域驅動設計）的核心思想：把業務邏輯獨立出來，跟技術細節隔離。

為什麼要 DDD？

不用 DDD（傳統 MVC）

業務邏輯散在 Controller 跟 Service 裡
跟資料庫、API 緊密耦合
換個 Vector DB 要改一堆地方

用 DDD

業務邏輯集中在 Domain Layer
用 interface 隔離技術實作
換 Vector DB 只動 Infrastructure

完整版 DDD 的五大元件

Entity（實體）

有唯一 ID 的物件，會隨時間變化但身份不變。例如：KnowledgeBase 知識庫，名字可以改，但 ID 永遠一樣。

Value Object（值物件）

沒 ID、不可變，靠值判斷相等。例如：EmbeddingVector 向量、ChunkSize 切塊大小。

Aggregate（聚合）

一群相關 Entity 的邊界，由一個「Aggregate Root」統一管理。例如：KnowledgeBase 是 Root，Document 是裡面的成員。

Repository（儲存庫）

Aggregate 的存取介面，但「實作」放在 Infrastructure。Domain 只看到介面，不知道用什麼 DB。

Domain Event（領域事件）

業務上「發生過」的事實。例如：DocumentIndexed 文件已索引完成，可以觸發後續動作（通知、統計）。

Domain Service（領域服務）

跨多個 Aggregate 的業務邏輯，不適合放在單一 Entity 上時用。例如：「在多個知識庫之間搜尋」。

⚠️ 別走火入魔

DDD 的目標是「讓業務邏輯清楚」，不是「每個檔案都要套模板」。簡單的 CRUD 不需要 DDD，複雜的業務（像我們這個多租戶 RAG）才值得。

🧠 3. 領域模型設計

針對你的需求（多知識庫 + 多租戶），先把領域模型畫出來，後面寫 code 才有依據。

Bounded Context（限界上下文）

把整個系統切成幾個獨立的「業務區塊」，每個區塊有自己的領域模型，互不干擾。

Tenant Context（租戶）
管理客戶、權限、配額
  • Tenant（Aggregate Root）：客戶資訊、訂閱方案、API 配額
  • ApiKey（Value Object）：API 金鑰、權限範圍
  • Quota（Value Object）：使用量限制
Knowledge Context（知識庫）★ 核心
管理文件、切塊、向量索引
  • KnowledgeBase（Aggregate Root）：知識庫名稱、所屬 Tenant、嵌入模型設定
  • Document（Entity）：文件內容、來源、更新時間
  • Chunk（Entity）：文件切出來的段落，含 EmbeddingVector
  • EmbeddingVector（Value Object）：向量陣列、維度、相似度計算
  • 領域事件：DocumentIndexed、KnowledgeBaseCreated
Chat Context（對話）
處理問答、檢索、生成回答
  • Conversation（Aggregate Root）：對話 session、屬於哪個 KnowledgeBase
  • Message（Entity）：問題、回答、引用來源
  • RetrievalResult（Value Object）：檢索結果（chunks + 相似度）
  • 領域事件：QuestionAnswered（可用來統計）
🔑 Aggregate 的邊界很重要

KnowledgeBase 跟 Document 都屬於同一個 Aggregate（KnowledgeBase 是 Root）。修改 Document 必須透過 KnowledgeBase，不能繞過。這保證業務規則的一致性（例如：刪除 KnowledgeBase 時，所有 Document 也跟著刪）。

🏗️ 4. 四層架構詳解

DDD 經典的四層架構：依賴方向永遠由外向內，Domain 層不依賴任何外部。

各層職責

Interface Layer
介面層 — 跟外界溝通
Controller / DTO / Guard
職責

接收 HTTP 請求 → 驗證 → 呼叫 Use Case → 回傳結果。不做業務邏輯。

前端類比

就像 React 的 page component，只負責「拿資料、傳資料」，不寫業務邏輯。

Application Layer
應用層 — 編排業務流程
Use Case / Command / Query
職責

協調多個 Domain 元件完成一個「使用案例」。例如「回答問題」這個 Use Case 要：呼叫向量搜尋 → 呼叫 LLM → 儲存對話紀錄。

關鍵原則
  • 每個 Use Case 一個 class，只做編排，不寫業務規則
  • 業務規則寫在 Domain，這層只是「叫」它們做事
Domain Layer ★ 核心
領域層 — 純業務邏輯
Entity / VO / Aggregate / Event
職責

所有業務規則、計算、驗證、不變條件。不依賴任何框架、不依賴 DB、不依賴外部 API。

關鍵原則
  • Repository 只定義 interface，實作放在 Infrastructure
  • Domain 應該可以「拔出來在另一個專案用」（極端的話）
  • 純 TypeScript，不用 NestJS 的 @Injectable（避免框架污染）
Infrastructure Layer
基礎設施層 — 技術實作
DB / Vector DB / API client
職責

實作 Domain 層定義的 Repository interface，串接 PostgreSQL、Qdrant、OpenAI 等外部系統。

關鍵原則
  • 把所有「換掉就要重寫」的東西放這層
  • Domain 想換 Vector DB？只要改這層，別處不動
🔑 依賴反轉原則（DIP）

傳統做法：Domain 直接呼叫 OpenAI API。問題：要換 Claude 就要改 Domain。
DDD 做法：Domain 定義 EmbeddingService interface，Infrastructure 實作 OpenAIEmbeddingService。要換 Claude 只要新增 ClaudeEmbeddingService，Domain 完全不動。

🏢 5. 多租戶設計

多租戶 = 一個系統服務多個客戶，每個客戶資料完全隔離。設計時要考慮三個關鍵：資料隔離、權限驗證、配額管理。

資料隔離策略：三選一

策略 說明 適合
共享 DB + tenant_id 所有租戶共用資料庫，每筆資料帶 tenant_id 過濾 大多數 SaaS（推薦）
共享 DB + Schema 隔離 同一個 DB，每個租戶獨立 schema 需要結構差異
獨立 DB 每個租戶獨立資料庫 金融、醫療等高敏感
💡 推薦：tenant_id 策略

對 RAG 系統來說，最實用的做法是：
• PostgreSQL：每張表都加 tenant_id 欄位
• Qdrant：用 metadata filter 做隔離（每個 vector 有 tenant_id payload）
• 更嚴格：每個 Tenant 用獨立的 Qdrant Collection

驗證 Tenant 的方式

前端每次請求帶 API Key，後端用 NestJS 的 Guard 在進入 Controller 之前先驗證。

tenant.guard.tsINTERFACE
@Injectable()
export class TenantGuard implements CanActivate {
  constructor(
    private tenantRepo: TenantRepository  // 注入 Domain 介面
  ) {}

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const req = context.switchToHttp().getRequest()
    const apiKey = req.headers['x-api-key']

    const tenant = await this.tenantRepo.findByApiKey(apiKey)
    if (!tenant) throw new UnauthorizedException()

    // 把 tenant 存進 request，後面 Use Case 取用
    req.tenant = tenant
    return true
  }
}

domain/knowledge/knowledge-base.aggregate.tsDOMAIN
// Aggregate Root：知識庫的入口
export class KnowledgeBase {
  private _documents: Document[] = []
  private _events: DomainEvent[] = []

  constructor(
    public readonly id: string,
    public readonly tenantId: string,    // 多租戶必備
    public name: string,
    public readonly embeddingModel: string,
  ) {}

  // 業務方法：加入文件（業務規則寫這裡）
  addDocument(content: string, source: string): Document {
    // 業務規則範例：同一個來源不能重複
    if (this._documents.some(d => d.source === source)) {
      throw new Error(`Document from ${source} already exists`)
    }

    const doc = new Document(crypto.randomUUID(), content, source)
    this._documents.push(doc)

    // 發出領域事件
    this._events.push(new DocumentAddedEvent(this.id, doc.id))
    return doc
  }

  get events(): DomainEvent[] {
    return [...this._events]
  }
}

Step 2：Application Layer（編排）

application/use-cases/ask-question.use-case.tsAPPLICATION
@Injectable()
export class AskQuestionUseCase {
  constructor(
    private kbRepo: KnowledgeBaseRepository,    // 注入 interface
    private vectorStore: VectorStoreRepository,
    private embeddingService: EmbeddingService,
    private chatService: ChatService,
  ) {}

  async execute(input: {
    tenantId: string,
    knowledgeBaseId: string,
    question: string,
  }) {
    // 1. 驗證知識庫存在且屬於該租戶
    const kb = await this.kbRepo.findById(
      input.knowledgeBaseId,
      input.tenantId
    )
    if (!kb) throw new NotFoundException('Knowledge base not found')

    // 2. 問題向量化
    const queryVector = await this.embeddingService.embed(
      input.question,
      kb.embeddingModel
    )

    // 3. 向量搜尋（自動帶 tenantId 隔離）
    const chunks = await this.vectorStore.search(
      kb.id,
      input.tenantId,
      queryVector,
      3
    )

    // 4. LLM 生成回答
    const answer = await this.chatService.complete({
      question: input.question,
      context: chunks.map(c => c.text).join('\n\n'),
    })

    return { answer, sources: chunks }
  }
}

Step 3：Infrastructure Layer（實作）

infrastructure/vector-store/qdrant.repository.impl.tsINFRA
@Injectable()
export class QdrantVectorStoreRepository extends VectorStoreRepository {
  private client: QdrantClient

  constructor(config: ConfigService) {
    super()
    this.client = new QdrantClient({
      url: config.get('QDRANT_URL')
    })
  }

  async search(
    knowledgeBaseId: string,
    tenantId: string,
    queryVector: EmbeddingVector,
    topK: number
  ): Promise<RetrievalResult[]> {
    // 重點：用 metadata filter 做多租戶 + 多知識庫隔離
    const results = await this.client.search('chunks', {
      vector: queryVector.toArray(),
      limit: topK,
      filter: {
        must: [
          { key: 'tenant_id',        match: { value: tenantId } },
          { key: 'knowledge_base_id', match: { value: knowledgeBaseId } },
        ]
      },
      with_payload: true,
    })

    return results.map(r => new RetrievalResult(
      r.payload.text as string,
      r.score
    ))
  }
}

常見陷阱

陷阱解法
Domain 層用了 NestJS 裝飾器 Domain 應該是純 TypeScript class，不要 @Injectable
Controller 直接呼叫 Repository 必須透過 Use Case，Controller 不該知道 Repository
Aggregate 太大 如果一個 Aggregate 包含上千個子 Entity，要拆開
沒做 tenant 隔離 所有 Repository 方法都要帶 tenantId 參數
Embedding 同步處理 大量文件用 Bull queue 背景跑，避免 API timeout

背景處理 Embedding（重要）

大量文件向量化會花很久時間。前端不該等。建議用 @nestjs/bull 把 Embedding job 丟到背景：

application/use-cases/index-document.use-case.ts
@Injectable()
export class IndexDocumentUseCase {
  constructor(
    @InjectQueue('indexing') private queue: Queue
  ) {}

  async execute(input: { tenantId, kbId, content }) {
    // 立刻回 200，背景跑
    await this.queue.add('embed-document', input)
    return { status: 'queued' }
  }
}
