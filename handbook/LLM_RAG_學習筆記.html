LLM 基礎原理

🔤
Token（詞塊）
語言的最小單位
定義 Token 是 AI 字典裡的「專屬 ID」。文字被切割成碎片，單一數字本身毫無意義，僅作為索引。例如：'Python' → ID: 492
BPE（Byte Pair Encoding）— OpenAI/Anthropic 採用 透過「頻繁子字串」進行合併。例如「我正在104人力銀行搜尋前端工程師職缺」原本 17 個字，BPE 後變成 5 個 Token（我正在 / 104人力銀行 / 搜尋 / 前端工程師 / 職缺），大幅壓縮。
OpenAI/Anthropic: BPE Google: Unigram
📐
Embedding（向量）
意義的數字化身
定義 將文字轉化為保留語意與關聯性的數字陣列（高達數千維度，如 OpenAI 的 1536 維）。範例：Python → [0.13, -0.27, 0.89, ...]
語意距離 — 餘弦相似度（Cosine Similarity） 電腦不懂「履歷」或「職缺」，但懂幾何數學。意義相近的詞在向量空間中距離極近。 例如搜尋「想找寫程式的工作」→ 系統找到：軟體工程師（高）、程式設計師（高）、行政助理（低）

RAG 架構詳解

⚠️
LLM 的兩大痛點
RAG 要解決的問題
痛點一：AI 幻覺（Hallucination） LLM 本質是「機率生成模型」。當缺乏特定知識時，會「一本正經地胡說八道」，產生看似合理卻錯誤的答案。
痛點二：企業資料斷層 資安壁壘：企業機密無法上傳公有雲訓練。時效性滯後：預訓練模型知識有截止日期，無法即時獲取最新資料。
🔍
RAG 核心概念
Retrieval-Augmented Generation — 以「開卷考試」重塑 AI
傳統 LLM
閉卷考
→
RAG
開卷考
R
Retrieval（檢索）

翻閱龐大的知識庫、職缺描述、薪資報告與產業趨勢分析

A
Augmented（增強）

將檢索出的相關段落與使用者原始問題，打包成富含背景知識的 Prompt

G
Generation（生成）

LLM 基於提供的正確資料，生成精準解答並附上引用標籤

🗄️
Vector Database（向量資料庫）
RAG 的核心基礎建設
關聯式資料庫（SQL）— 傳統方法的問題 依賴精確字面比對。搜尋「前端工程師」找不到「Web Developer」或「React 開發者」。面對職稱多樣性極度脆弱。
向量資料庫 — 語意近似度搜尋 依賴 ANN（近似最近鄰）搜尋。能將「前端工程師」與「Web Developer」、「React 開發者」等語意關聯起來，實現精準跨職稱媒合。
Pinecone Milvus Qdrant Elasticsearch

RAG 系統建立流程
Indexing（索引建立）四步驟
1
資料提取（Data Parsing）

將非結構化資料（履歷、職缺描述）及結構化資料（薪資範圍、地區）萃取為純文本，保留 Metadata（職務類別、工作年資）

2
語意分塊（Chunking）

將長文切分為適當大小的片段（Chunks）。必須設定重疊區（Overlap）避免語意斷裂

3
向量化（Embedding）

呼叫 Embedding 模型將 Chunk 轉為高維度矩陣

4
儲存（Vector Storage）

將向量與原始文本存入 Vector DB，建立 ANN 索引以支援快速語意搜尋

RAG 評估三元組（RAG Triad）
如何衡量 RAG 系統成效
指標評估重點反例
上下文關連性
Context Relevance		 檢索的資料是否真的相關？ 搜尋「Java 資深工程師」卻回傳大量「行政助理」履歷
事實一致性

答案相關性
Answer Relevance		 是否精準解決用戶提問？ 問「中年轉職建議」，系統回傳「新鮮人面試技巧」
🚨
RAG 常見失敗節點
四大問題點
1. 知識庫缺失（Missing Content） 企業文件本無答案時，系統可能產生錯誤。需建立「優雅拒答」機制。
2. 檢索遺漏（Missed Top Ranked） 演算法門檻或 Chunking 切割導致語意破碎，遺漏相關文件。
3. 上下文迷失（Not in Context） 塞入太多 Context 超過 Token 限制，LLM 在大量雜訊中迷失重點。
4. 格式錯誤（Format Error） LLM 沒有依指令生成特定格式（如 JSON），導致後端系統錯誤。
🤖
AI Agent（智慧代理）
從被動知識庫到主動思考
RAG 只是起點 未來企業應用不再只是「你問我答」，AI Agent 能主動規劃與執行任務。
🧠 大腦（LLM） 負責邏輯推理與任務拆解，分析複雜職缺描述與求職者資料
💾 記憶（RAG） 透過向量庫精準提取歷史知識，檢索面試題庫、薪資數據、公司評論
🦾 手與腳（Tools/API） 呼叫內部系統執行動作：自動安排面試、發送通知、更新應徵狀態

1Token 與 Embedding 的主要差別是什麼？
A
Token 有語意，Embedding 只是索引
B
Token 是整數 ID（無語意），Embedding 是高維浮點數向量（有語意）
C
兩者是完全相同的概念
D
Embedding 是 Token 的子集合
✅ 正確！Token 是語言的最小單位（索引），Embedding 是將 Token 轉為保留語意的高維向量。
❌ 再想想！Token = 整數 ID（如 Python → 492），Embedding = 語意向量（如 [0.13, -0.27, ...]）。
2RAG 的三個字母分別代表什麼？
A
Reasoning, Analysis, Generation
B
Retrieval, Automation, Generation
C
Retrieval（檢索）, Augmented（增強）, Generation（生成）
D
Ranking, Aggregation, Generation
✅ 正確！RAG = 檢索增強生成，先從知識庫檢索相關內容，再增強 Prompt，最後讓 LLM 生成回答。
❌ RAG 是 Retrieval-Augmented Generation（檢索增強生成）！
3BPE（Byte Pair Encoding）的主要作用是？
A
將文字轉為向量空間
B
透過合併頻繁子字串來壓縮 Token 數量
C
計算兩個向量的餘弦相似度
D
訓練語言模型的損失函數
✅ 正確！BPE 透過頻繁子字串合併來壓縮，讓「我正在104人力銀行搜尋前端工程師職缺」(17字) 變成 5 個 Token。
❌ BPE 是 Tokenizer 方法，透過合併頻繁的子字串（如「前端工程師」→ 1 個 Token），大幅壓縮 Token 數量。
4下列哪個選項正確描述了「模型蒸餾」？
A
直接複製大模型的參數到小模型
B
將模型的精度從 32-bit 降低到 8-bit
C
讓小模型學習大模型的思考方式與輸出分布（模仿）
D
用人工標註資料微調現有模型
✅ 正確！蒸餾是讓 Student 模型模仿 Teacher 模型的思維與機率分布，而非複製參數。
❌ 蒸餾不是複製參數，而是讓小模型（Student）學會大模型（Teacher）如何思考和判斷！
5RAG 評估三元組（RAG Triad）包含哪三個指標？
A
精確率、召回率、F1 分數
B
速度、成本、準確率
C
上下文關連性、事實一致性、答案相關性
D
索引品質、向量維度、模型大小
✅ 正確！Context Relevance（檢索的對嗎？）+ Groundedness（有沒有幻覺？）+ Answer Relevance（解決問題了嗎？）
❌ RAG Triad = 上下文關連性 + 事實一致性（Groundedness）+ 答案相關性，用來全面評估 RAG 系統品質。
6為什麼向量資料庫優於傳統 SQL 資料庫用於語意搜尋？
A
向量資料庫速度更快
B
向量資料庫存儲成本更低
C
向量資料庫支援語意近似度搜尋（ANN），而非只是字面比對
