📊 RAG 整體流程圖

RAG 分為兩個獨立階段：建立知識庫（一次性，像 build）和查詢回答（runtime，像 fetch API）。

PHASE 1 — 建立知識庫（跑一次，像 npm run build）
PHASE 2 — 查詢回答（每次提問觸發，像 fetch API）
核心觀念

Phase 1 把所有文件變成「向量座標」，Phase 2 把問題也變成座標，找出座標距離最近的段落，再交給 LLM 看著回答。整個流程的關鍵是：用同一個 Embedding 模型處理文件和問題，這樣才能在同一個座標空間比對。

📦 Library 完整說明

所有要用到的套件詳細介紹：用途、為什麼選它、有沒有替代方案。

openai
官方 OpenAI Node.js SDK
必裝 v4.x+
做什麼用

呼叫 OpenAI 的兩個關鍵 API：embeddings（文字 → 向量）和 chat.completions（送 prompt 拿回答）。整個 RAG 都靠它。

為什麼選它
  • 官方維護，更新最即時，bug 少
  • Type 完整，TypeScript 友善
  • 支援 streaming，可以做出 ChatGPT 那種逐字輸出效果
替代方案
  • @anthropic-ai/sdk：用 Claude 模型替代 GPT
  • @google/generative-ai：用 Gemini 模型替代
  • ollama：本地跑開源模型，省 API 費用
@langchain/textsplitters
LangChain 文字分塊工具
必裝 輕量
做什麼用

把長文件切成適合大小的小段（Chunks）。提供 RecursiveCharacterTextSplitter（最常用）、MarkdownTextSplitter（按標題切）、TokenTextSplitter（按 Token 數切）等。

為什麼選它
  • 業界標準，幾乎所有 RAG 教學都用這個
  • 智能切割，會盡量保留段落、句子的完整性
  • 支援 overlap，避免語意被切斷
替代方案
  • llm-chunk：更輕量、設定更少，適合快速 PoC
  • semantic-chunking：用語意相似度切（最聰明但較慢）
@qdrant/js-client-rest
Qdrant 向量資料庫的 Node.js 客戶端
必裝 REST
做什麼用

連線到 Qdrant Vector DB，做向量的存入（upsert）和搜尋（search）。Qdrant 是專門儲存向量、做相似度搜尋的資料庫。

為什麼選 Qdrant
  • 開源免費，本地用 Docker 就能跑起來
  • 效能好，Rust 寫的，速度快
  • 有 Web UI，方便 debug 看資料
  • 支援 metadata filter，可以在搜尋時加條件
替代方案（Vector DB）
  • @pinecone-database/pinecone：純雲端服務，零維運，有免費方案
  • chromadb：另一個熱門開源選擇
  • weaviate-ts-client：功能豐富但較重
  • @supabase/supabase-js + pgvector：如果你已經在用 Supabase
pdf-parse
PDF 文字抽取工具
看需求
做什麼用

讀取 PDF 檔案，把裡面的文字抽出來變成純字串。如果你的知識庫只有 Markdown 或純文字，可以不用。

替代方案
  • mammoth：處理 Word 檔（.docx）
  • cheerio：抓 HTML 內文（像 jQuery 語法）
  • node-html-parser：另一個輕量 HTML 解析器
  • unstructured API：可以一次處理 PDF/Word/HTML/PPT 等格式（雲端服務）
dotenv
環境變數管理
必裝
做什麼用

讀取 .env 檔案裡的環境變數（像 API Key），避免把敏感資訊寫死在程式碼裡。前端應該很熟悉，跟 Vite/Next.js 的 .env 機制一樣。

注意
  • 記得把 .env 加到 .gitignore，不要 commit 到 GitHub
  • Node.js 20.6+ 內建 --env-file flag，新版可以不裝這套件
express
Web 框架（看你要不要做成 API）
看需求
做什麼用

把 RAG 包成 HTTP API，讓前端 React/Vue app 可以呼叫。如果只是寫腳本測試，不用裝。

替代方案
  • fastify：效能更好、現代化的選擇
  • hono：輕量、能跑在 Edge Runtime（Cloudflare Workers）
  • Next.js API Routes：如果你已經在用 Next.js，直接用 API routes 就好

PHASE 1 — 建立知識庫（跑一次）

🔨 1. 讀檔 → 切塊 → 向量化 → 存庫

STEP 1

讀取文件

前端類比

就像 fs.readFileSync() 讀靜態資源，再交給 parser 處理

RAG 用途

無論是 PDF、Markdown 還是 HTML，都要先抽出純文字才能後續處理

src/index.js
import fs from 'fs'
import pdfParse from 'pdf-parse'
import dotenv from 'dotenv'
dotenv.config()

// 讀 Markdown（最簡單）
function readMarkdown(filePath) {
  return fs.readFileSync(filePath, 'utf-8')
}

// 讀 PDF
async function readPDF(filePath) {
  const buffer = fs.readFileSync(filePath)
  const data = await pdfParse(buffer)
  return data.text
}

const text = readMarkdown('./docs/company-faq.md')
console.log(`讀取完成，共 ${text.length} 個字元`)
STEP 2

Chunking — 把長文切成小段

⚠️ 為什麼要切？

LLM 有 Token 上限（像 API request body 大小限制）。整篇文件丟進去太貴又不準確，切成小段後每段獨立做語意比對，搜尋更精準。

src/index.js
import { RecursiveCharacterTextSplitter } from '@langchain/textsplitters'

async function chunkText(text) {
  const splitter = new RecursiveCharacterTextSplitter({
    chunkSize: 500,    // 每段最多 500 字元
    chunkOverlap: 50, // 前後重疊 50 字元（避免語意斷裂）
  })
  return splitter.splitText(text)
}

const chunks = await chunkText(text)
console.log(`切成 ${chunks.length} 段`)
💡 chunkSize 怎麼選？

一般建議 300~500 字元。太小失去上下文，太大搜尋準確度下降。如果文件是 Q&A 格式，可以一個問答 = 一段（用 MarkdownTextSplitter 按標題切）。

STEP 3

Embedding — 每段文字轉向量

前端類比

把每個元素轉成它在頁面上的 { x, y, width } 座標

RAG 用途

把文字轉成 1536 維的數字陣列，「語意相近」的段落距離近

src/index.js
import OpenAI from 'openai'

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY })

async function getEmbedding(text) {
  const res = await openai.embeddings.create({
    model: 'text-embedding-3-small', // 便宜又夠用
    input: text,
  })
  return res.data[0].embedding  // 1536 維陣列
}

async function embedAllChunks(chunks) {
  const results = []
  for (const [i, chunk] of chunks.entries()) {
    const vector = await getEmbedding(chunk)
    results.push({ id: i, text: chunk, vector })
    await new Promise(r => setTimeout(r, 100)) // 避免 rate limit
    console.log(`進度：${i+1}/${chunks.length}`)
  }
  return results
}

const embeddedChunks = await embedAllChunks(chunks)
✅ 模型選擇

text-embedding-3-small：1536 維，便宜（$0.02 / 1M tokens），大多場景夠用。text-embedding-3-large：3072 維，精度更高，貴 6 倍，搜尋準確度要求高再用。

STEP 4

存入 Qdrant Vector DB

前端類比

Collection = SQL 的 Table，upsert = INSERT OR UPDATE

RAG 用途

把向量 + 原始文字 + metadata 一起存。查詢時靠向量找，顯示時用原文

src/index.js
import { QdrantClient } from '@qdrant/js-client-rest'

const qdrant = new QdrantClient({ url: process.env.QDRANT_URL })
const COLLECTION = 'my-knowledge-base'

async function setupCollection() {
  const exists = await qdrant.collectionExists(COLLECTION)
  if (!exists.exists) {
    await qdrant.createCollection(COLLECTION, {
      vectors: {
        size: 1536,           // 對應 text-embedding-3-small
        distance: 'Cosine',   // 餘弦相似度
      }
    })
  }
}

async function saveToVectorDB(embeddedChunks) {
  const points = embeddedChunks.map(chunk => ({
    id: chunk.id,
    vector: chunk.vector,
    payload: {
      text: chunk.text,         // 原始文字（給 LLM 讀）
      source: 'company-faq',
      createdAt: new Date().toISOString(),
    }
  }))
  await qdrant.upsert(COLLECTION, { points })
  console.log(`存入 ${points.length} 筆向量！`)
}

// 主流程
await setupCollection()
await saveToVectorDB(embeddedChunks)
console.log('✅ 知識庫建立完成！')
PHASE 2 — 查詢回答（每次提問觸發）

🔍 2. 查詢與回答

STEP 5

搜尋相關段落

💡 重要

查詢時要用同一個 Embedding 模型把問題轉成向量，才能在同一個座標空間裡比對。如果用不同模型，向量「語言」不通，搜尋結果會完全錯亂。

src/query.js
import OpenAI from 'openai'
import { QdrantClient } from '@qdrant/js-client-rest'
import dotenv from 'dotenv'
dotenv.config()

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY })
const qdrant = new QdrantClient({ url: process.env.QDRANT_URL })
const COLLECTION = 'my-knowledge-base'

export async function searchRelevantChunks(question, topK = 3) {
  // 1. 問題轉向量（注意：要用同一個模型！）
  const res = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: question,
  })
  const questionVector = res.data[0].embedding

  // 2. Vector DB 找最近 Top-K
  const results = await qdrant.search(COLLECTION, {
    vector: questionVector,
    limit: topK,
    with_payload: true,
  })

  // 3. 取出原文
  return results.map(r => ({
    text: r.payload.text,
    score: r.score,           // 相似度分數，可以用來判斷信心度
  }))
}
STEP 6

組合 Prompt，讓 LLM 回答

⚠️ Prompt 是關鍵

沒寫好 Prompt 的話，LLM 還是會自己編造答案。要明確告訴它「只能根據資料回答」，找不到就說不知道。

src/query.js
export async function askWithRAG(question) {
  // 1. 撈相關段落
  const chunks = await searchRelevantChunks(question, 3)
  const context = chunks.map((c, i) => `[資料 ${i+1}]\n${c.text}`).join('\n\n')

  // 2. 組 Prompt（這段超關鍵）
  const prompt = `你是公司的 AI 助理，請嚴格根據以下參考資料回答問題。
規則：
1. 只能根據參考資料回答，不要自己編造
2. 如果資料中沒有相關資訊，誠實說「我在現有資料中找不到答案」
3. 回答結尾標註你引用了哪些段落（例：根據 [資料 1]）

【參考資料】
${context}

【問題】
${question}`

  // 3. 送給 LLM
  const completion = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [{ role: 'user', content: prompt }],
    temperature: 0.3,        // 低溫讓回答更穩定
  })

  return {
    answer: completion.choices[0].message.content,
    sources: chunks,            // 回傳出處
  }
}

// 測試
const result = await askWithRAG('公司遠距工作的規定？')
console.log('回答：', result.answer)
console.log('引用段落：', result.sources)
✅ 大功告成

LLM 現在只會根據你的知識庫回答，不會亂猜。問到沒收錄的資訊會誠實說「找不到」。

🌐 3. 包成 API（給前端呼叫）

OPTION

用 Express 包成 REST API + Streaming

src/server.js
import express from 'express'
import cors from 'cors'
import { askWithRAG, searchRelevantChunks } from './query.js'

const app = express()
app.use(cors())
app.use(express.json())

// 一般 API：等回答完整再回傳
app.post('/api/ask', async (req, res) => {
  try {
    const { question } = req.body
    const result = await askWithRAG(question)
    res.json(result)
  } catch (err) {
    res.status(500).json({ error: err.message })
  }
})

// Streaming API：逐字回傳（像 ChatGPT）
app.post('/api/ask/stream', async (req, res) => {
  const { question } = req.body
  const chunks = await searchRelevantChunks(question)
  const context = chunks.map(c => c.text).join('\n\n')

  res.setHeader('Content-Type', 'text/event-stream')
  res.setHeader('Cache-Control', 'no-cache')

  const stream = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    stream: true,
    messages: [{
      role: 'user',
      content: `根據資料回答：\n${context}\n\n問題：${question}`
    }]
  })

  for await (const chunk of stream) {
    const token = chunk.choices[0]?.delta?.content ?? ''
    if (token) res.write(token)
  }
  res.end()
})

app.listen(3001, () => {
  console.log('🚀 RAG API 啟動於 http://localhost:3001')
})

前端怎麼呼叫 Streaming

frontend (React)
async function askStream(question, onToken) {
  const res = await fetch('/api/ask/stream', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ question })
  })

  const reader = res.body.getReader()
  const decoder = new TextDecoder()

  while (true) {
    const { done, value } = await reader.read()
    if (done) break
    const token = decoder.decode(value)
    onToken(token)  // 逐字更新 UI
  }
}

// React 用法
const [answer, setAnswer] = useState('')
askStream('公司有什麼福利？', (token) => {
  setAnswer(prev => prev + token)  // 一個個 token 累加
})

💡 4. 常見問題 & 調校技巧

問題 原因 解法
回答不準確 Chunk 切太小，上下文不足 增加 chunkSize 到 600~800，或增加 topK 到 5
回答太慢 topK 塞太多，或模型太大 用 gpt-4o-mini，減少 topK，開 streaming
AI 還在亂猜 Prompt 沒限制它 明確寫「找不到就說不知道」、降低 temperature 到 0.2
語意被切斷 chunkOverlap 太小 增加到 80~100
文件更新後沒反應 Vector DB 沒重建 重跑 index.js，或做增量更新
API 費用太高 Embedding 重複呼叫 建立後快取向量，只有新文件才重新 embed
搜尋結果不相關 Embedding 模型對中文不夠好 改用 text-embedding-3-large 或多語模型 BGE-M3
進階方向

掌握基礎後，建議再研究這些主題提升 RAG 品質：Metadata Filter（搜尋加條件）、Hybrid Search（語意 + 關鍵字混合）、Re-ranking（搜尋結果二次排序）、Query Rewriting（自動改寫使用者問題提升準確度）。
