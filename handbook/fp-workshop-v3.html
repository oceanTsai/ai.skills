0115 分鐘 · 觀念導入

為何需要
函數式編程？

// why functional programming · the motivation

大多數 bug 不是來自演算法錯誤，而是來自「狀態在錯誤的時間被錯誤的人改了」。函數式編程，本質上是在處理這個問題。

i核心觀念 · 程式碼 = 資料 + 動作
命令式 vs 函數式：兩種根本不同的世界觀

命令式編程（imperative）的核心是「動作的序列」——告訴 CPU 一步一步該做什麼：建立變數、更新變數、判斷條件、跳轉。它的世界裡，程式的本質是「狀態」，而程式碼是「改變狀態的指令」。

函數式編程的核心是「資料的轉換」——告訴電腦你要從一個資料變成另一個資料，中間怎麼走由抽象的「函數」負責。它的世界裡，程式的本質是「表達式」，而程式碼是「把輸入映射成輸出的數學公式」。

命令式像是一張「食譜步驟」（先切菜、再下鍋、再翻炒）；函數式則像是一條「輸送帶」（原料 → 清洗機 → 切割機 → 包裝機 → 成品）。
為什麼前端工程師特別需要 FP？

前端工程的本質就是「把資料變成 UI」。React 的核心公式 UI = f(state) 就是純函數的概念——同樣的 state，永遠渲染出同樣的 UI。Redux、Zustand、Vue 的響應式系統、RxJS、TanStack Query——這些工具的設計哲學全都建立在 FP 之上。

不學 FP，你只是在「用」這些工具；學了 FP，你才真正理解這些工具為什麼這樣設計、以及該怎麼正確使用。

前端工程師的日常痛點

請先回憶你維護專案時遇到的這些情況：

  • 「咦？這個 state 怎麼變了？是誰改的？」
  • 巢狀 if/else 與重複的 null 檢查
  • 同樣輸入卻得到不同結果（時序、共享狀態作祟）
  • 不敢動既有函數，因為「改了會壞掉哪裡」沒人知道
  • 非同步程式碼像金字塔，錯誤處理散落各處

這些問題的共同根源：副作用沒有被馴服、資料隨時可變、控制流程隱含在指令中。

兩種風格的對照

▲ Imperative · 命令式

告訴電腦怎麼做。建立變數、迴圈、條件、修改值。關注「步驟」與「狀態變化」。

● Declarative · 宣告式

告訴電腦要什麼。用 map、filter、reduce 描述資料如何被轉換。關注「轉換」與「資料流」。

javascript // imperative傳統寫法
// 找出所有成年使用者的姓名，並轉成大寫
const result = [];
for (let i = 0; i < users.length; i++) {
  if (users[i].age >= 18) {
    result.push(users[i].name.toUpperCase());
  }
}
javascript // declarativeFP 風格
// 同樣需求，但描述的是「資料轉換」
const result = users
  .filter(u => u.age >= 18)
  .map(u => u.name.toUpperCase());

FP 帶來的四個核心好處

i
可預測
純函數同樣輸入永遠同樣輸出，debug 從「猜」變成「推」。
ii
可組合
小函數像樂高一樣組合，比類別繼承靈活百倍。
iii
可測試
沒有副作用就不需要 mock，單元測試輕量到不可思議。
iv
可並行
不可變資料天然避開競態，Web Worker、SSR 都更穩。
▸ 講師提醒

不要把 FP 講成「比 OOP 更高級」。FP 與 OOP 是不同的工具，前端工作中我們其實兩者並用——元件結構偏向組件導向，狀態管理與資料轉換則高度 FP。我們學 FP 是為了補齊另一半工具箱。

Programs must be written for people to read, and only incidentally for machines to execute. — Harold Abelson · SICP
0225 分鐘 · 核心觀念

純函數與
副作用

// pure functions & side effects

如果整個函數式編程只能學一個觀念，那就是純函數。其他所有概念都是它的延伸，或為了配合它而生。

?什麼是「純」？為什麼數學家發明這個概念？
從數學的「函數」說起

在數學課，f(x) = x + 1 是一個函數。給它 3，永遠得到 4；給它 100，永遠得到 101。它不會看心情，不會今天回 4，明天回 5。也不會在算完之後偷偷把你家的時鐘往前撥一小時。這就是「純」。

但程式設計裡的「函數」一開始不是這樣的。傳統 C 語言裡，函數可以讀寫全域變數、讀寫檔案、改變 caller 傳進來的指標指向的記憶體。一個叫 calculate 的函數可能順便把資料庫改了。名字叫函數，但行為不像函數。

純函數的兩個條件

條件一：相同輸入 → 相同輸出。不管何時、何地、被誰呼叫，只要 input 一樣，output 必定一樣。不可以依賴 Date.now()、Math.random()、外部全域變數。

條件二：沒有副作用。函數內部不能對「函數參數與回傳值」以外的東西造成任何改變——不能修改傳進來的物件、不能寫 DOM、不能發 API、不能 console.log。

純函數像「計算機」——你按 2 + 3，它顯示 5；不管你按多少次、什麼時候按，結果都一樣，計算機也不會偷偷打電話給你媽。
f(x) = y · 同樣的 x，永遠得到同樣的 y

什麼是「副作用」？

!副作用 · side effect
正式定義

當一個函數除了「接收輸入、回傳輸出」之外，還對函數外部產生了任何可觀察的影響，這影響就叫「副作用」（side effect）。

常見的副作用清單

● 修改全域變數或外部變數
● 修改傳入的參數（mutating arguments）
● 操作 DOM（document.getElementById、appendChild...）
● 網路請求（fetch、WebSocket）
● 讀寫 localStorage、cookies、檔案
● console.log、alert、彈出視窗
● 拋出例外（throw）
● 依賴或修改時間（Date.now()）、亂數（Math.random()）

副作用為什麼可怕？

副作用本身不是錯的——一個沒有副作用的程式什麼也做不了（連 console.log 都不能用）。真正可怕的是「不可見的副作用」：當你看到 calculatePrice(item) 這個函數，你以為它只算個價錢，但它其實悄悄改了購物車、發了 API、改了使用者狀態——那才是 bug 的溫床。

FP 不消滅副作用，FP 是把副作用「可見化、集中化、推到邊界」。

不純的常見元凶

javascript // impure ✗這些都是不純的
// ❌ 依賴外部變數
let tax = 0.05;
function priceWithTax(p) { return p * (1 + tax); }

// ❌ 修改傳入的物件 (mutating argument)
function addItem(cart, item) { cart.push(item); return cart; }

// ❌ 依賴隨機 / 時間
function expireAt(min) { return Date.now() + min * 60000; }

// ❌ I/O 副作用
function save(user) { return fetch('/api', { method: 'POST', body: user }); }
javascript // pure ✓純化後的版本
// ✅ 把依賴變成參數（顯式注入）
function priceWithTax(p, rate) { return p * (1 + rate); }

// ✅ 回傳新物件，不改原本的
function addItem(cart, item) { return [...cart, item]; }

// ✅ 把時間當參數注入
function expireAt(now, min) { return now + min * 60000; }

「把副作用推到邊界」是什麼意思？

★架構心法 · functional core, imperative shell
核心 / 殼層 結構

想像你的程式像一顆雞蛋。蛋黃（核心）是你的業務邏輯：計算、轉換、決策——這部分要全部保持純粹。蛋白（殼層）是與外界互動的層：fetch、寫 DOM、讀 localStorage——副作用都集中在這裡。

就像核能電廠的「反應爐」與「控制室」：反應爐裡的物理規律是純的（同樣條件、同樣結果），但你要靠控制室的閥門、感應器（副作用）才能與它互動。
React / Redux 是這個哲學的標準範例

● reducer 是純函數：(state, action) => newState，永遠不能 fetch、不能改 state
● selector 是純函數：從 state 推導 UI 資料
● useEffect 是副作用區：API、訂閱、計時器都放這
● component 是純函數：props => UI

看出來了嗎？整個 React 生態就是 FP 思維在 UI 框架的落地。

外部世界 → 殼層（不純） → 核心（純） → 殼層（不純） → 外部世界

練習 2.1 · 純函數辨識

下列哪些是純函數？為什麼？

javascript // identify思考題
const a = (x, y) => x + y;
const b = (arr) => arr.sort();
const c = (arr) => [...arr].sort();
const d = (user) => ({ ...user, name: user.name.trim() });
const e = (id) => document.getElementById(id);
const f = (x) => Math.sqrt(x);

解答：a、c、d、f 是純函數。b 會修改原陣列（sort 是 mutating）；e 依賴 DOM 這個外部狀態。

0315 分鐘 · 進階觀念

引用透明性

// referential transparency

引用透明聽起來像學術名詞，但它其實是「你為什麼能信任純函數」的數學表達。理解它，你會獲得一種新的閱讀程式碼的能力。

=引用透明 · referential transparency
白話定義

一段程式碼如果是「引用透明」的，意思是：你可以用它的回傳值，直接取代它本身，整個程式的行為完全不變。

舉例：如果 add(2, 3) 永遠回傳 5，那程式裡所有 add(2, 3) 都可以直接改寫成 5，行為一模一樣。這個函數就是引用透明的。

為什麼這個性質這麼重要？

因為它意味著——當你讀程式碼時，你可以用「等號替換」來理解任何一段程式，就像國中數學課做代換題一樣。這是極為強大的「推理工具」。

就像化學裡的「守恆律」——當你知道 2H₂ + O₂ = 2H₂O 永遠成立，你就可以放心做計算，不用擔心今天的水分子會偷偷變成別的東西。
反例：不透明的函數

看這段 code：let counter = 0; const next = () => ++counter;

next() 第一次回 1、第二次回 2、第三次回 3。你不能把 next() 替換成任何固定值，因為它每次的結果不同。所以 next 不是引用透明的——它的回傳值依賴外部隱藏狀態。

引用透明 ≈ 純函數，但更精確

嚴格說，「引用透明」是描述「表達式」的性質，「純函數」是描述「函數」的性質。但實務上你可以把它們當同一回事。每個純函數的呼叫都是引用透明的；每個引用透明的呼叫背後都是純函數。

用「等號替換」來推理程式

引用透明帶來的最大實務好處：你可以把任何純函數呼叫，替換成它的結果，程式碼意義完全不變。這聽起來理所當然，但仔細想想，命令式程式碼根本做不到這件事。

javascript // substitution引用透明的威力
const double = x => x * 2;
const addOne = x => x + 1;

const result = addOne(double(5));
// 因為 double 和 addOne 都是純函數，我可以這樣推理：
// addOne(double(5)) === addOne(10) === 11
// 整段在程式裡可以「安全地」用 11 取代
javascript // not transparent不透明的對照
let count = 0;
const tick = () => ++count;

tick() + tick();
// 第一個 tick() = 1，第二個 tick() = 2，結果是 3
// 你不能把它改寫成 tick() + 2 或 2 + tick()，結果都會不一樣
// 因為 tick 不是引用透明的

引用透明帶來的三大實務能力

i
記憶化（memoization）
因為同輸入一定同輸出，可以放心把結果快取起來，下次直接查表。React 的 useMemo / useCallback 就建立在這上面。
ii
惰性求值（lazy evaluation）
既然替換等價，那就可以延後執行——什麼時候真的需要值，再去算。Haskell 的整個語言都建立在這上面。
iii
平行化
純函數之間沒有共享狀態，可以隨意分配到不同執行緒、Worker、機器上跑，不用煩惱鎖的問題。
▸ React 工程師的福音

React 的 useMemo、useCallback、React.memo、Reselect 的 createSelector——這些優化工具都要求你傳純函數。原因正是引用透明：只有引用透明的函數才能放心地「同輸入就回傳上次的結果」。當你寫了不純的 selector，整個記憶化就會出 bug。

0425 分鐘 · 核心觀念

不可變性

// immutability

「我又沒改原本的東西，怎麼還是壞了？」九成的狀態管理 bug，是因為某處不小心改到了共享的物件。不可變性是純函數的雙胞胎兄弟。

∞不可變性 · immutability
不可變性是什麼？

不可變（immutable）的意思是：一個值一旦被建立，就永遠不能被改變。你想要「修改」？沒辦法。你只能「以舊值為基礎，建立一個新的值」。

在 JS 裡，原始型別（number、string、boolean、null、undefined、symbol、bigint）本來就是不可變的。"hello".toUpperCase() 不會改變原本的 "hello"，它建立一個新的字串 "HELLO"。但物件和陣列預設是可變的，這就是麻煩的開始。

為什麼可變性是 bug 的溫床？

當一個物件可以被任何人在任何時候修改，你就必須在心裡同時追蹤所有可能修改它的地方。當這個物件被傳給三個函數、又被存進兩個快取，誰是真正的擁有者？誰可以改？什麼時候改？——這就是「shared mutable state」的本質惡夢。

可變物件像「公共白板」——任何人都能改任何東西，最後沒人知道為什麼上面寫的是這個。不可變物件像「已發布的文件」——要改？發新版，舊版永遠在那不變。
不可變性與純函數的關係

它們是互為因果的：一個函數要「純」，就不能修改傳進來的物件（否則就是副作用）；如果整個系統的資料都是不可變的，函數就自動不可能意外修改它們，於是更容易保持純粹。兩者一起運作才是 FP 的完整體驗。

JavaScript 的陷阱：傳參考

⚠淺拷貝 vs 深拷貝 · 一個經典坑

JS 的物件與陣列是傳參考（pass by reference）的。意思是：當你把一個物件當參數傳給函數、或賦值給另一個變數，傳的不是物件本身，而是「指向那個物件的記憶體位置」。所以兩個變數可能指向同一個物件，改其中一個會影響另一個。

spread operator（...）只是淺拷貝——它只複製第一層，巢狀的物件還是共用同一個參考。這是新手最常踩的坑。

javascript // gotcha淺拷貝陷阱
const user = { name: 'Alice', profile: { age: 30 } };
const copy = { ...user };           // 淺拷貝
copy.profile.age = 99;             // 也改到了 user.profile.age！
console.log(user.profile.age);    // 99 ❗

三種更新方式的權衡

A · Spread / 手動展開

javascript // spread細粒度更新
const updated = {
  ...user,
  profile: { ...user.profile, age: 31 }
};

適合淺層結構，深層巢狀會變地獄。

B · structuredClone（原生 API）

javascript // es2022+原生深拷貝
const updated = structuredClone(user);
updated.profile.age = 31;       // 改 updated 不會影響 user

深拷貝、原生支援、效能不錯，但函數與類別實例（Date 以外）會掉資訊。

C · Immer（推薦給 React / Redux）

javascript // with immerdraft 模式
import { produce } from 'immer';

const updated = produce(user, draft => {
  draft.profile.age = 31;        // 看起來像改 mutable，實際產生新物件
});

程式碼像 mutable，結果卻是 immutable，是工程實務上的最佳平衡。Immer 內部用了 Proxy + 結構共享（structural sharing），只複製真的改到的部分，效能很好。

▲ 會改原陣列（mutating）

push · pop · shift · unshift · splice · sort · reverse

● 回傳新陣列（non-mutating）

map · filter · concat · slice · flat · toSorted · toReversed · with

下面的 reducer 為何讓 Redux state 不更新 UI？

javascript // bugredux 不更新
function updateUser(state, action) {
  state.users[action.id].name = action.name;
  return state;
}

提示：state 與 state.users 的參考都沒變。React 用 Object.is 判斷 state 是否變化，比的是參考（reference）而非內容。改成回傳全新的物件樹即可。這也是為什麼不可變性是 React 渲染機制的基石。

0510 分鐘 · 語法觀念

為何 FP 慣用
箭頭函數？

// arrow functions · why they fit FP

很多人以為箭頭函數只是「簡寫」。但 FP 大量使用它，是因為箭頭函數的設計，剛好對齊純函數的本質。

→箭頭函數的三個 FP 友善特性
第一：表達式優先（expression-oriented）

傳統 function 是「陳述句」（statement）——你執行它做事情。箭頭函數則是「表達式」（expression）——它是一個值。

當你寫 x => x + 1，整個東西就是一個值，可以放進變數、陣列、參數、任何地方。這呼應了 FP 的核心信念：函數是值（first-class）。語法本身就在告訴你「我是一個值」。

function 像「食譜」（一份做事的說明書）；箭頭函數像「數學公式」（直接是個關係表達式）。
第二：沒有自己的 this

傳統 function 有自己動態綁定的 this，會根據呼叫方式（直接呼叫、當方法呼叫、用 call、new 呼叫）改變。這個性質在 OOP 中有用，但在 FP 中是純粹的麻煩——因為 FP 函數不應該依賴隱藏的「呼叫上下文」。

箭頭函數沒有自己的 this，它從外層繼承（詞法綁定）。換句話說，它對「我在哪被呼叫」一無所知——這完美對應「函數只該依賴它的參數」的純函數原則。

第三：簡潔讓組合可讀

當你要寫 pipe(filter(isAdult), map(toName), sortBy(getAge)) 這種組合時，每個小函數越短越好。u => u.age >= 18 比 function (u) { return u.age >= 18; } 在組合裡可讀得多。

關鍵不是「字少」，而是「視覺噪音少」——函數的本體（邏輯）佔比例越高，意圖越清楚。

語法對照

javascript // arrow vs function語法對比
// 傳統 function
function double(x) {
  return x * 2;
}

// 箭頭函數（單參數 / 單表達式）
const double = x => x * 2;

// 多參數
const add = (a, b) => a + b;

// 多行（用大括號）
const process = (data) => {
  const cleaned = data.trim();
  return cleaned.toUpperCase();
};

// 回傳物件要包小括號（否則 {} 會被當成 block）
const wrap = x => ({ value: x });

this 行為的差異

javascript // this binding關鍵差異
const obj = {
  name: 'Alice',
  greetFn: function() { return `Hi, ${this.name}`; },
  greetArrow: () => `Hi, ${this?.name}`
};

obj.greetFn();      // 'Hi, Alice'（this = obj）
obj.greetArrow();   // 'Hi, undefined'（this 來自外層作用域）

何時不用箭頭函數？

×
物件方法
需要 this 指向物件時，用傳統 function（或 ES2015 method shorthand）。
×
建構子
箭頭函數不能用 new，因為它沒有 [[Construct]]。
×
需要 hoisting
function 宣告會 hoist，箭頭函數是 const，要先宣告再用。
×
需要 arguments
箭頭函數沒有 arguments 物件，要用 rest（...args）。
▸ 結論

FP 偏好箭頭函數，不只是「省字」，而是因為箭頭函數的設計（表達式、無 this、簡潔）剛好符合純函數哲學。寫 FP 的時候用箭頭函數，是讓語法配合思維，而不只是省鍵盤。

0625 分鐘 · 核心觀念

高階函數與
一等公民

// higher-order & first-class functions

在 JavaScript 中，函數可以像數字、字串一樣被傳遞、儲存、回傳。這個性質是整個 FP 的基礎，沒有它，後面所有觀念都不存在。

★「一等公民」是什麼意思？
「first-class」的詞源

這個說法來自 1960 年代，電腦科學家 Christopher Strachey 提出。所謂「一等」（first-class），指的是某類東西在語言中擁有完整的權利：可以被命名、可以被當參數、可以被當回傳值、可以放進資料結構。

在 C 語言裡，數字是一等公民（你可以做以上所有事），但函數不是（你不能把函數放進陣列）。在 JS、Python、Haskell 裡，函數也是一等公民——它跟數字、字串享有同等地位。

那「高階函數」呢？

高階函數（higher-order function, HOF）是指：函數的「輸入或輸出」是另一個函數的函數。

● arr.map(fn) 接收函數當參數 → map 是 HOF
● const greet = name => () => \`Hi, ${name}\` 回傳函數 → greet 是 HOF

為什麼 HOF 這麼重要？

因為它讓你能「把行為當成資料來傳遞」。傳統 OOP 要把行為傳遞，必須先定義介面、再寫類別、再實例化、再傳實例——非常繁瑣。FP 直接 arr.map(x => x * 2)，行為就傳過去了。

OOP 的「策略模式」是「為了傳一個行為，蓋了一整棟房子」；HOF 是「行為直接從口袋掏出來」。

函數作為值的四種用法

javascript // first-class函數的四種用法
// 1. 存進變數
const greet = (name) => `Hi, ${name}`;

// 2. 存進陣列 / 物件
const handlers = { onClick: greet, onHover: console.log };

// 3. 當參數傳遞
['Alice', 'Bob'].map(greet);

// 4. 當回傳值
const makeGreeter = (prefix) => (name) => `${prefix}, ${name}`;

三大基石：map / filter / reduce

▣從心智模型認識這三個方法
map：一對一轉換

給 map 一個函數，它會把這個函數套用到陣列每一個元素上，回傳長度一樣的新陣列。原始資料的「結構」（長度、順序）不變，只有「內容」變了。

filter：取子集合

給 filter 一個「斷言函數」（predicate，回傳 true/false），它會留下回傳 true 的元素。新陣列的長度可能變短，但每個元素本身不變。

reduce：摺疊（fold）

reduce 是三者中最強的。它把陣列「摺疊」成單一值——這個值可以是數字、物件、甚至另一個陣列。map 和 filter 都可以用 reduce 寫出來，但反過來不行。

map 是「每張照片加濾鏡」、filter 是「只留下喜歡的照片」、reduce 是「把所有照片拼成一張拼貼畫」。

map · 一對一轉換

javascript // map
const prices = [100, 200, 300];
const withTax = prices.map(p => p * 1.05);
// [105, 210, 315]

filter · 取子集合

javascript // filter
const orders = [{ paid: true }, { paid: false }, { paid: true }];
const paid = orders.filter(o => o.paid);
// [{ paid: true }, { paid: true }]

reduce · 折疊成單一值

javascript // reduce最強大的一個
// 加總
[1,2,3].reduce((acc, x) => acc + x, 0);  // 6

// 分組 (groupBy)
users.reduce((acc, u) => {
  (acc[u.role] ??= []).push(u);
  return acc;
}, {});

// reduce 是萬能的，map 和 filter 都能用它寫
const myMap = (arr, f) => arr.reduce((acc, x) => [...acc, f(x)], []);
▸ 別過度使用 reduce

reduce 雖然強，但程式碼可讀性常常輸給 map+filter 鏈。實務原則：能用 map/filter 描述清楚就別用 reduce。只有當你真的需要把多項摺成一項（總和、計數、分組、組裝）才用 reduce。

關鍵心智：函數作為「策略」

HOF 讓你把行為當參數傳。同一個資料管線，換掉一個函數就改變整體行為——這就是策略模式的本質，但比 OOP 簡潔太多。

javascript // strategy injection行為注入
// 通用排序：把比較策略當參數注入
const sortBy = (keyFn) => (arr) =>
  [...arr].sort((a, b) => keyFn(a) - keyFn(b));

const byAge   = sortBy(u => u.age);
const byScore = sortBy(u => -u.score);  // 加負號就是降冪

0725 分鐘 · 核心觀念

函數組合

// function composition

如果純函數是 FP 的原子，那組合就是 FP 的化學鍵。把小函數串在一起，是 FP 構建大型程式的方式。

∘什麼是函數組合？
數學起源

函數組合的概念源自數學：給定兩個函數 f 和 g，它們的「組合」f ∘ g 定義為「先做 g，再把結果丟給 f」，也就是 (f ∘ g)(x) = f(g(x))。

這在程式裡有什麼用？因為它讓你能「把小函數黏起來，當成大函數來用」。你不必預先寫一個大函數，而是用小積木拼出大行為。這就是 FP 構建大型程式的核心方式——不是繼承、不是組合（OOP 意義的）、是函數組合。

為什麼這個比繼承還好？

OOP 的繼承是「固定的關係」：A 繼承 B，編譯時就決定了。FP 的組合是「動態的關係」：你想怎麼組合就怎麼組合，組合方式本身也可以動態決定。

繼承像「家族族譜」（你的爸是誰寫死了）；函數組合像「樂高積木」（你想拼飛機就拼飛機，想拼船就拼船）。
為什麼小函數比大函數好？

● 容易測試：每個小函數只做一件事，測試案例簡單
● 容易命名：行為單一，名字就好取
● 容易重用：你今天寫的 trim，明年的專案還能用
● 容易修改：要加一步？在組合鏈裡插一個進去，其他不動

(f ∘ g)(x) = f(g(x))

在程式裡，我們有兩種風格寫組合：

compose · 從右往左讀（數學派）

javascript // compose
const compose = (...fns) => (x) => fns.reduceRight((v, f) => f(v), x);

const shout = compose(
  s => s + '!',
  s => s.toUpperCase(),
  s => s.trim()
);
shout('  hello  '); // 'HELLO!'

pipe · 從左往右讀（工程派，更直覺）

javascript // pipe
const pipe = (...fns) => (x) => fns.reduce((v, f) => f(v), x);

const shout = pipe(
  s => s.trim(),
  s => s.toUpperCase(),
  s => s + '!'
);
' hello ' → trim → toUpperCase → + '!' → 'HELLO!'
▸ 為什麼 pipe 更受歡迎？

因為「讀程式碼像讀句子」。pipe(trim, toUpper, addBang) 的閱讀順序就是執行順序，符合人類由左至右的直覺。Lodash 提供 _.flow、Ramda 是 R.pipe、最新提案的 Pipeline Operator（|>）也是同樣概念。

組合的威力：把資料管線顯式化

▶從「動詞」到「名詞」的思維轉變

命令式程式碼像在說「動詞」：do this, then do that。每個動詞分散在不同的程式碼行，整體流程是隱式的，必須讀完才能拼湊出來。

用 pipe 寫的程式碼，整個流程變成一個名詞：「使用者輸入處理管線」、「報表生成管線」、「資料同步管線」。你可以幫整條管線命名，整條管線本身可以被傳遞、組合、覆寫。

這是巨大的轉變——你不再是「在某個位置寫一段邏輯」，而是「把一條管線當成一個物件去設計」。Redux 的 middleware、RxJS 的 operator chain、Next.js 的 middleware pipe，都是這個思維的應用。

javascript // real pipeline真實場景
// 處理使用者輸入 → 驗證 → 標準化 → 存檔
const processUserInput = pipe(
  trimAll,
  validate,
  normalize,
  toServerShape,
  save
);

// 想多加一步「打日誌」？插進去就好
const processUserInput = pipe(
  trimAll,
  validate,
  tap(console.log),   // tap = 偷看不改值
  normalize,
  toServerShape,
  save
);

tap：除錯用的小工具

javascript // tapdebugging in pipeline
const tap = (fn) => (x) => { fn(x); return x; };

// 在 pipeline 任何一處塞 tap(console.log) 都能看中間值
// tap 接收一個副作用函數，執行它，但回傳原值（不改變資料流）

柯里化

// currying & partial application

把「需要三個參數的函數」變成「連續呼叫三次、每次傳一個參數」，這就是柯里化。它讓函數變得可以「半成品傳遞」，是組合 pipeline 的關鍵。

λ什麼是柯里化？為什麼要這樣折磨自己？
命名由來

「柯里化」（currying）以邏輯學家 Haskell Curry 命名（同一個人，Haskell 語言也以他命名）。這個技巧的數學基礎更早，由 Moses Schönfinkel 在 1924 年提出，後來 Curry 將它系統化。

柯里化的核心轉換

把「一次接收 n 個參數」的函數，轉成「每次接收 1 個參數、回傳新函數」的函數鏈。直到所有參數都收齊，才回傳最終結果。

● 原本：add(1, 2, 3) 回傳 6
● 柯里化後：add(1)(2)(3) 也回傳 6
● 中間狀態：add(1) 回傳「等著收第二個參數」的新函數
● 再中間：add(1)(2) 回傳「等著收第三個參數」的新函數

為什麼這樣有用？

因為你可以提前綁定一部分參數，留下「半成品函數」之後用。這就是「函數工廠」的概念——同樣的母函數，不同的部分綁定，產出一系列專用函數。

柯里化像「客製化咖啡機」——你不必每次都重新告訴它「我要拿鐵、燕麥奶、少糖、加熱」。你先設定好「我的拿鐵」（部分綁定），之後只要按一下，就出你的拿鐵。
為什麼配合 pipe 特別好用？

pipe 要求裡面每個函數都是「單參數函數」（一進一出）。但現實中你很多函數是多參數的（如 map(fn, arr)）。柯里化讓你能「先綁定 fn，留下要收 arr 的單參數函數」，這就能塞進 pipe 了。

f(a, b, c) ⇨ f(a)(b)(c)
javascript // curry從多參數到柯里化
// 一般版
const add = (a, b, c) => a + b + c;
add(1, 2, 3);              // 6

// 柯里化版
const addC = a => b => c => a + b + c;
addC(1)(2)(3);              // 6
const add5 = addC(2)(3);     // add5(x) = x + 5（半成品！）

為什麼這很有用？實務範例

當函數柯里化後，你可以提前綁定部分參數，得到一個「配置好的、更小的函數」。這就是「函數工廠」。

javascript // function factory配置 → 專用函數
// 把設定凍結起來，產生專用函數
const log = level => tag => msg =>
  console.log(`[${level}] ${tag}: ${msg}`);

const error = log('ERROR');              // 凍結 level
const apiError = error('API');         // 凍結 tag
apiError('Timeout');                // [ERROR] API: Timeout

「資料放最後」的設計慣例

⇄data-last vs data-first

觀察兩個寫法：

● data-first（lodash 預設）：_.map(arr, fn)——資料在前
● data-last（Ramda、lodash/fp）：R.map(fn, arr)——資料在後

為什麼 FP 函數庫一致選擇 data-last + 柯里化？因為當資料放最後，你可以先綁定「怎麼處理」，留下「處理什麼」當參數。這個「半成品」剛好就是 pipe 想要的「單參數函數」。

反過來，data-first 的設計，每次都要先給資料，柯里化的好處就消失了。所以 data-last 不是任意選擇，是為了配合 pipe / compose 的必然結果。

javascript // data-last為什麼這樣設計
// data-last + 柯里化 = 完美與 pipe 配合
const map    = f => arr => arr.map(f);
const filter = p => arr => arr.filter(p);

const processUsers = pipe(
  filter(u => u.active),    // 先綁定 predicate，留下「等收 arr」的半成品
  map(u => u.email)         // 同上
);

processUsers(users);  // 漂亮的資料管線

柯里化 vs 偏應用

柯里化是「強制變成一次一個參數」；偏應用（partial application）則是「一次綁定幾個都可以」。在 JS 中常用 .bind 或自寫 helper：

javascript // partial application
const partial = (fn, ...presets) => (...rest) => fn(...presets, ...rest);

const request = (baseURL, path, options) => fetch(baseURL + path, options);
const api = partial(request, 'https://api.example.com');
api('/users', { method: 'GET' });
▸ 何時用、何時不用

柯里化在「同一個函數要被多次配置成不同變體」時超好用（HTTP client、logger、formatter、與 pipe 配合）；但日常單次呼叫的函數不必柯里化，會讓 call site 變吵。柯里化是工具，不是儀式。

0925 分鐘 · 進階觀念

Functor
是什麼？

// the mappable container

當我們說「Functor」時，大多數人其實每天都在用它——只是不知道它有名字。陣列就是 Functor，Promise 也是 Functor。

□從「容器」開始理解 Functor
問題：你不只想處理「值」，還想處理「裝著值的東西」

真實世界裡，我們很少直接處理一個赤裸的值。我們處理的是：

● 陣列裡的值
● Promise 裡（將來會到達的）值
● 可能為 null 的值
● 可能是錯誤 的值
● 樹狀結構某節點上的值

這些「裝著值的東西」叫做「容器」（container 或 context）。問題是：怎麼處理容器裡的值，而不打開容器？

Functor 的答案：定義一個 map 方法

Functor 就是一個有 map 方法的容器，這個 map 方法做兩件事：

● 接收一個函數 f
● 把容器裡的值套用 f，結果還是裝在同型別的容器裡

關鍵在於最後一點——map 不會把值「倒出來」。你 map 一個陣列得到陣列；你 map 一個 Promise 得到 Promise；你 map 一個 Maybe 得到 Maybe。容器型別永遠不變，只有裡面的內容被轉換。

Functor 像「真空包裝食物」——你想把生菜變成沙拉？你不必拆開包裝（那會搞髒）。你把整個包裝丟進「沙拉加工機」（map），出來的還是真空包裝，但裡面的內容變成沙拉了。
Functor = 一個有 map 方法的容器

當一個「盒子」滿足以下兩條，它就是 Functor：

  1. 它包裝了某個值（可以是任意型別）
  2. 它有一個 map(f) 方法：把 f 套用到裡面的值，回傳一個同型別的新盒子

陣列就是 Functor

javascript // array as functor
[1, 2, 3].map(x => x * 2);
// [2, 4, 6] —— 依然是陣列，只是內容變了

自己動手寫一個 Box

javascript // minimal functor最小可運作的 Functor
const Box = (x) => ({
  map: (f) => Box(f(x)),
  fold: (f) => f(x),       // 把值拿出來
  inspect: () => `Box(${x})`
});

Box('  hello  ')
  .map(s => s.trim())
  .map(s => s.toUpperCase())
  .fold(s => s + '!');
// 'HELLO!'

注意 map 永遠回傳新的 Box，所以可以鏈式呼叫。這跟陣列 .map().filter().map() 是完全一樣的模式。

Promise 也是 Functor

javascript // promise as functor
fetch('/api/user')
  .then(r => r.json())     // .then 在 Functor 觀點下就是 map
  .then(u => u.name);

// 包在 Promise 裡的值，透過 then 被轉換，仍然包在 Promise 裡

為什麼 Functor 重要？

i
統一介面
陣列、Promise、Maybe、Tree 都用 map，不用為每個型別記不同 API。
ii
保留上下文
map 不會把值「倒出來」，能安全地在容器內持續轉換。
iii
延遲執行
Promise / Observable 的 map 可以串接，但執行時機由容器決定。
▸ Functor 法則（簡述）

真正的 Functor 要滿足兩條法則：恆等性（box.map(x => x) 等於原本的 box）與組合性（box.map(f).map(g) 等於 box.map(x => g(f(x)))）。日常使用不必背，但若你發現某個 map 不符合這兩條，那它就不是真正的 Functor（這也是為什麼 Promise 嚴格說不是「乾淨的」Functor，因為它的 then 還會 unwrap）。

1030 分鐘 · 進階觀念

Monad
不再神秘

// monad · maybe / either

「Monad 是一個 endofunctor 範疇中的 monoid」——這句話沒幫到任何人。讓我們用 JS 工程師的語言來理解：Monad 就是「會自動展開盒子的 Functor」。

⊕從 Functor 的痛點，理解 Monad 為什麼存在
Functor 解決了「在容器內處理值」

有了 Functor，我們可以對 Box、陣列、Promise 裡的值做轉換，而不必把值倒出來。很美好。

但很快就遇到新問題：巢狀容器

當你 map 的函數本身也回傳一個容器時，就會出現「容器套容器」的情況。比方說 findUser(id) 回傳 Box(user)，那 Box(1).map(findUser) 就會回傳 Box(Box(user))——巢狀了。

實務上這超常見——當你的轉換步驟本身會「可能失敗」、「可能 null」、「需要非同步」時，這個函數一定回傳容器。每多一步就多一層 Box，最後變成 Box(Box(Box(...)))，沒人玩得起來。

Monad 的解法：flatMap

Monad 是強化版的 Functor，多了一個 flatMap 方法（也叫 chain、bind、then）。它跟 map 幾乎一樣，但會自動把巢狀的容器攤平一層。

● map(f)：f 回傳 普通值，Box 層數不變
● flatMap(f)：f 回傳 Box(值)，自動攤平成 Box(值)

如果 Functor 是「真空包裝食物」（map 不打開包裝），那 Monad 就是會處理「套疊包裝」的版本——當你的加工機回傳的也是真空包裝食物，flatMap 不會讓你變成「包裝套包裝」，它會幫你自動拆掉一層。
為什麼這這麼重要？

因為它讓你能把「可能失敗的、有副作用的、非同步的」運算，組合成一條乾淨的管線。沒有 Monad 時你要寫 try/catch、null check、巢狀 then；有了 Monad，這些都消失，剩下純粹的資料轉換。

Monad 的本質：讓「會生成新容器」的函數能與「在容器內處理」的世界協作。

Functor 的痛點：盒子套盒子

javascript // problem
// 假設 findUser 回傳 Box(user)
const result = Box(1)
  .map(id => findUser(id));        // Box(Box(user)) ❌

Monad 的解法：flatMap

javascript // flatmapmonad 的核心方法
const Box = (x) => ({
  map:     (f) => Box(f(x)),
  flatMap: (f) => f(x),                // 不再包一層！
  fold:    (f) => f(x)
});

Box(1)
  .flatMap(id => findUser(id))    // Box(user)
  .flatMap(u => findPosts(u.id)); // Box(posts) ✓
▸ 你早就在用 Monad

Promise 的 .then 同時具備 map 與 flatMap 的能力——它會自動 unwrap 嵌套的 Promise，這就是為什麼 then(x => fetch(...)) 不會變成 Promise<Promise<...>>。陣列的 flatMap 同理：[1,2,3].flatMap(x => [x, x*2]) 自動把 [[1,2],[2,4],[3,6]] 攤平成 [1,2,2,4,3,6]。

三種實用的 Monad

Maybe · 處理 null / undefined

javascript // maybe monad取代 ?. 與 ??
const Some = (x) => ({
  map: (f) => Some(f(x)),
  flatMap: (f) => f(x),
  getOrElse: () => x,
  isNone: false
});

const None = {
  map: () => None,
  flatMap: () => None,
  getOrElse: (d) => d,
  isNone: true
};

const Maybe = (x) => (x == null ? None : Some(x));

// 不再需要層層 ?. 與 ??
const name = Maybe(user)
  .map(u => u.profile)
  .map(p => p.name)
  .map(n => n.toUpperCase())
  .getOrElse('Anonymous');

Either · 處理錯誤而不丟例外

javascript // either monad取代 try / catch
const Right = (x) => ({
  map: (f) => Right(f(x)),
  flatMap: (f) => f(x),
  fold: (_l, r) => r(x)
});

const Left = (e) => ({
  map: () => Left(e),
  flatMap: () => Left(e),
  fold: (l, _r) => l(e)
});

const safeDiv = (a, b) =>
  b === 0 ? Left('Cannot divide by zero') : Right(a / b);

safeDiv(10, 2)
  .map(x => x + 1)
  .fold(
    err => `錯誤: ${err}`,
    val => `結果: ${val}`
  );  // '結果: 6'

為什麼這比 try/catch 好？

▲ try / catch

錯誤是隱式的（看不出哪個函數會丟），破壞函數的純粹性，難以組合。例外會「跨越」呼叫棧，違反引用透明。

● Either

錯誤是型別的一部分，TypeScript 編譯器會強制你處理，且可以與 pipe 完美組合。每個函數的成功/失敗都顯式可見。

心智圖：Functor vs Monad

F
Functor
map(f)：f 回傳普通值，容器層數不變。例：陣列、Promise。
M
Monad
flatMap(f)：f 回傳同型別的容器，會自動攤平。例：Maybe、Either、Promise。

API 模組

// final workshop · refactor a real api layer

前面學的所有觀念，在這一章會用一個你每天都會寫的場景串起來：呼叫 API、驗證資料、處理錯誤、轉換成 UI 需要的格式。

情境：使用者個人頁面

需求：從 API 取得使用者，過濾啟用中的、根據註冊日期排序，並轉換成 UI 需要的格式。失敗時要有清楚的錯誤。

命令式版本（典型寫法）

javascript // before典型寫法
async function getActiveUsers() {
  try {
    const res = await fetch('/api/users');
    if (!res.ok) throw new Error('API error');

    const users = await res.json();
    const result = [];

    for (const u of users) {
      if (u.active) {
        result.push({
          id: u.id,
          name: (u.firstName + ' ' + u.lastName).trim(),
          since: new Date(u.createdAt).getFullYear()
        });
      }
    }

    result.sort((a, b) => b.since - a.since);
    return result;
  } catch (e) {
    console.error(e);
    return [];
  }
}

FP 重構版本

javascript // afterFP 風格
// ── 小工具（純函數）──
const pipe = (...fns) => (x) => fns.reduce((v, f) => f(v), x);
const prop = (k) => (o) => o[k];
const filter = (p) => (xs) => xs.filter(p);
const map = (f) => (xs) => xs.map(f);
const sortBy = (keyFn) => (xs) => [...xs].sort((a, b) => keyFn(b) - keyFn(a));

// ── 領域函數（純）──
const isActive   = (u) => u.active;
const fullName   = (u) => `${u.firstName} ${u.lastName}`.trim();
const yearOf     = (u) => new Date(u.createdAt).getFullYear();
const toViewModel = (u) => ({ id: u.id, name: fullName(u), since: yearOf(u) });

// ── 資料管線（組合上面所有小函數）──
const transformUsers = pipe(
  filter(isActive),
  map(toViewModel),
  sortBy(prop('since'))
);

// ── 副作用（推到邊界）──
async function getActiveUsers() {
  const res = await fetch('/api/users');
  if (!res.ok) throw new Error('API error');
  const users = await res.json();
  return transformUsers(users);
}

重構帶來了什麼？

  • 核心邏輯（transformUsers）完全是純函數，可以單獨測試，不需要 mock fetch。
  • 副作用只剩一處：fetch 與錯誤處理集中在 getActiveUsers 入口（核心 / 殼層架構）。
  • 每個小函數都可以單獨重用：fullName、yearOf 在其他地方也用得到。
  • 修改容易：要加篩選條件？在 pipe 裡插一行；要換排序？換一個 keyFn。

進階版：用 Either 處理錯誤

javascript // with either無 try / catch 的錯誤處理
const safeFetch = async (url) => {
  try {
    const res = await fetch(url);
    return res.ok
      ? Right(await res.json())
      : Left(`HTTP ${res.status}`);
  } catch (e) {
    return Left(`Network: ${e.message}`);
  }
};

const result = (await safeFetch('/api/users'))
  .map(transformUsers)
  .fold(
    err => ({ ok: false, error: err }),
    data => ({ ok: true, data })
  );
