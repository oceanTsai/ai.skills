Token & Embedding

🔤
Token 是什麼？
AI 語言的最小單位
前端你熟悉的

你呼叫 API 時，資料是 JSON 格式傳輸的。字串 "前端工程師" 在底層其實是一串 UTF-8 bytes，電腦才能處理。

AI 世界的概念

LLM 也看不懂「中文字」，所以先把文字切成 Token（詞塊），再轉成 整數 ID。這就像 UTF-8 encoding，只是 AI 版本。

// 前端呼叫 API，你熟悉的樣子 const text = "我正在找前端工程師職缺" // Tokenizer 把它切成「詞塊」並編成 ID // 類比：就像 string → charCodeAt()，但更聰明 tokenize(text) // → ["我正在", "找", "前端工程師", "職缺"] // → [ 101, 234, 3892, 561 ] ← Token ID // BPE 演算法：高頻字組會被合併成單一 Token // "前端工程師" 常一起出現 → 整個變 1 個 Token！ // 效果：17 個字 → 壓縮成 4~5 個 Token ✨
💡 為什麼要壓縮？ LLM 有「Token 上限」（就像 API request body 有大小限制），壓縮越好，能處理的文字越長、費用越低。
📐
Embedding 是什麼？
把「意思」轉成數字陣列
⚠️ Token 的問題 Token ID 只是索引，本身沒有意義。101 和 102 差 1，不代表它們語意相近。
前端你熟悉的

你用 CSS 的 座標系定位元素。top: 20px; left: 50px; 這兩個數字讓你知道元素「在哪裡」、跟其他元素的「距離」。

AI 世界的概念

Embedding 就是把每個詞放進 高維度座標空間（1536 維）。意思相近的詞，座標就距離近——「前端工程師」跟「Web Developer」在這空間裡幾乎是鄰居。

// 類比：CSS 是 2D 座標，Embedding 是 1536D 座標 // position: { top: 20, left: 50 } ← 2 維 const embedding_前端工程師 = [0.82, -0.31, 0.67, ...1533 個數字...] const embedding_WebDeveloper = [0.79, -0.28, 0.71, ...1533 個數字...] const embedding_行政助理 = [-0.4, 0.55, -0.2, ...1533 個數字...] // 計算「餘弦相似度」(cosineSimilarity) ← 就像算兩點距離 cosineSimilarity(前端工程師, WebDeveloper) // → 0.94 很近 ✅ cosineSimilarity(前端工程師, 行政助理) // → 0.11 很遠 ❌
✅ 這有什麼用？ 搜尋「前端工程師」→ 系統透過 Embedding 距離，也能找到「Web Developer」、「React 開發者」，不再只是死板的關鍵字比對！

Transformer 原理

⚙️
Transformer 是 LLM 的核心引擎
Encoder（理解）+ Decoder（生成）
前端你熟悉的

前端有 編譯器（Compiler）把你寫的 TypeScript 或 JSX 「理解並轉譯」成瀏覽器看得懂的 JS。

AI 世界的概念

Transformer 的 Encoder = 理解輸入文字的意思（像讀懂 TS 語法）；Decoder = 根據理解逐字生成回答（像輸出 JS）。

模型類型 前端類比 擅長 代表模型

Decoder（GPT/LLM） Copilot——看到上文，預測並生成下一行 生成文字、對話、翻譯 GPT、Claude、Gemini
💡 前端角度理解 LLM 生成原理 LLM 每次只預測「下一個 Token」，就像 autocomplete。輸入「<div class="」→ 預測下一個 token 可能是「container」(70%)、「wrapper」(15%)、「main」(8%)… 然後一直重複，直到生成完整回答。
🧮
Encoder 的核心：位置編碼 + Attention
機器怎麼「讀懂」一句話
📍 位置編碼（Positional Encoding） LLM 要同時知道「這個詞是什麼意思」跟「它在句子哪個位置」。就像 React 的 key prop——光知道元素是什麼不夠，還要知道它在 list 哪個位置。
前端你熟悉的

event delegation：一個父元素監聽所有子元素的事件，根據 event.target 判斷是誰觸發，並給予不同權重。

Attention 機制

處理「活動吧有活動的活動吧」時，第二個「活動」需要「注意」前後哪些詞，才能判斷它是「聚會」還是「運動」。Attention 就是這個加權判斷機制。

RAG 架構

🤔
為什麼需要 RAG？
LLM 的兩個致命傷
前端你熟悉的情境

你的 React app 沒有串 API，全用 hardcode 假資料。資料永遠是舊的、跟後端脫節，使用者看到錯誤資訊。

LLM 面對的問題

LLM 的知識是訓練時「燒進去」的，沒辦法即時更新。問它公司最新政策？它要嘛不知道，要嘛瞎掰（幻覺）。

⚠️ 痛點 1：AI 幻覺 LLM 猜不到答案時，會「一本正經地亂說」——就像前端 undefined 卻不報錯，直接渲染了錯誤的值。
⚠️ 痛點 2：知識過時 模型訓練有截止日，公司最新 FAQ、最新規定它根本不知道——就像 bundle 打包後就定版，沒辦法 hot update。
🔍
RAG 怎麼解決？
Retrieval-Augmented Generation — 讓 LLM 「開卷考試」
前端你熟悉的

加了 API fetch！使用者查詢 → 前端即時呼叫 API → 拿到最新資料 → 渲染正確內容。不再依賴 hardcode。

RAG 的做法

使用者提問 → 先去 向量資料庫撈最相關的文件 → 把文件塞進 Prompt → LLM 根據「真實資料」回答，不再瞎猜。

// 沒有 RAG（像 hardcode） askLLM("公司遠距工作規定是什麼？") // LLM: "根據一般企業規範，遠距工作通常…" ← 瞎猜！幻覺出現 😱 // 有 RAG（像 fetch 最新資料） const docs = await vectorSearch("遠距工作規定") // 撈相關文件 const prompt = `根據以下文件回答：\n${docs}\n\n問題：公司遠距工作規定？` askLLM(prompt) // LLM: "根據公司 2024 年 HR 手冊第 3 條…" ← 有依據！✅
RAG 完整流程（前端視角）
1
建立知識庫 Index（一次性，像 build 流程）

把公司文件 → 切成小段（Chunking）→ 每段轉成 Embedding 向量 → 存進 Vector DB。就像 webpack build，只要文件更新才需要重跑。

2
使用者提問（Runtime，像 API request）

把問題也轉成 Embedding → 在 Vector DB 找「距離最近」的文件段落（ANN Search）→ 取出 Top-K 筆。

3
Prompt 增強（像組 request body）

把撈到的文件段落 + 原始問題 → 組成一個完整 Prompt → 送給 LLM。

4
LLM 生成回答（像 API response）

LLM 只根據塞進去的文件回答，不再瞎猜，還可以附上出處。

AI Agent — RAG 的進化版
從「查資料」到「主動執行任務」
前端你熟悉的

純 fetch data + 渲染 的靜態頁面，升級為有 useEffect + 多個 API + 狀態管理 的互動 App。

Agent 的概念

不只是「問答」，AI Agent 可以自己規劃步驟、呼叫工具（API）、執行動作（發 email、更新資料庫），最後給你結果。

🧠 大腦（LLM）

負責推理與規劃，就像 App 的 business logic，決定「下一步做什麼」。

💾 記憶（RAG）

從向量庫撈歷史知識，就像 App 的 state + cache，不用每次重問。

🦾 工具（Tools/API）

呼叫外部系統，就像前端呼叫後端 API——發通知、查資料庫、排行程。

🔄 Agentic Loop

規劃 → 執行 → 觀察結果 → 反思調整 → 再執行。像 while loop，直到任務完成。

模型訓練與微調

🔄
三階段訓練流程
從一片空白到聰明的 AI
前端你熟悉的

一個新人工程師的成長路徑：先廣泛自學（看書、教程）→ 進公司學專案規範（帶教）→ 在實戰中被 code review 磨練（強化學習）

AI 訓練流程

Pretrain（海量文本自學）→ SFT（人工標註資料微調）→ RLHF（人類回饋強化學習）

1
預訓練 Pretrain — 「自學階段」

用海量無標註文本（整個網路）訓練。模型學會語言規則，但不知道怎麼對話。就像新人工程師自學程式，能寫 code 但不懂公司流程。
▸ 成本極高（需要超大算力），不是一般公司能做的事

2
SFT 監督微調 — 「帶教階段」

用人工標註的 Q&A 對話資料，教模型「正確的回答方式」。就像 Tech Lead 帶著新人做 code review，糾正壞習慣。
▸ 成本中等，需要高品質標註資料

3
RLHF 強化學習 — 「實戰磨練」

讓模型生成多個回答，人類評分哪個更好，模型從回饋中學習。就像工程師從 code review 留言、user feedback 中持續進步。
▸ 讓模型從「能用」變成「聰明」

🎯
Fine-tuning（微調）是什麼？
幫模型「專精化」
前端你熟悉的

你 fork 了一個 open source UI library，根據公司設計系統改了 color token、加了公司專用 component，客製化成公司版本。

Fine-tuning

拿現成的大模型（如 Gemma3、LLaMA），用公司自己的 FAQ 資料繼續訓練它，讓它「專精」公司業務知識，說話更符合公司風格。

💡 LoRA / QLoRA — 高效微調技術 不更新整個模型（太貴），只更新一小部分「增量參數」（LoRA 矩陣）。就像不重寫整個 library，只 override 你需要的幾個 CSS variables——省時省錢，效果幾乎一樣。
🧬
模型蒸餾（Knowledge Distillation）
讓小模型學大模型的「思考方式」
前端你熟悉的

Tree-shaking：webpack 把龐大的 npm package 打包時，只保留你實際用到的程式碼，輸出更小的 bundle。效能差不多，但體積大幅縮減。

模型蒸餾

讓小模型（Student）模仿大模型（Teacher）的「思考分布」，而不是複製參數。輸出更小、更快的模型，能力幾乎保留。

// 例子：DeepSeek R1 (671B 大模型) 蒸餾成小模型 // 就像把一個 50MB 的 bundle → tree-shake 成 3MB Teacher: DeepSeek-R1 (671B 參數，很強但很重) ↓ 蒸餾 Student: DeepSeek-R1-Distill-Qwen-7B (7B 參數，輕量快速) // 不是複製參數，是讓小模型學大模型 // 「這個問題，我猜 A 的機率是 0.7，B 是 0.2，C 是 0.1」 // 的這種「機率分布思維」

三種方法怎麼選？

📊
Prompt vs RAG vs Fine-tuning
用前端開發決策類比

💬 Prompt Engineering

前端類比：直接改 CSS inline style

快速、彈性，不需要額外設定。但可能被其他樣式蓋掉（不穩定），Token 長度有限（就像 inline style 太長很難維護）。

成本 ⭐⭐⭐ 速度 ⭐⭐⭐
適合場景 PoC 原型、一次性任務、簡單問答

🔍 RAG

前端類比：串 API，即時取最新資料

知識即時更新（不需重新訓練），大幅減少幻覺，可以附上來源出處。需要維護向量庫和檢索品質。

成本 ⭐⭐⭐⭐ 準確 ⭐⭐⭐⭐
適合場景 企業知識庫、客服問答、需要最新資料的系統

🏋️ Fine-tuning

前端類比：客製化 fork 整個 UI library

模型深度「記住」領域知識，推論快又穩定。但前期訓練成本極高，知識更新需要重新訓練。

成本 ⭐ 速度 ⭐⭐⭐⭐⭐
適合場景 金融、醫療等垂直領域，大規模產品化服務
💡 課程最佳實踐：RAFT = RAG + Fine-tuning 先用 Fine-tuning 讓模型有「語感」和「專業知識基礎」，再加上 RAG 確保知識隨時是最新的。就像先讀書考到執照，工作時再查最新的法規手冊。
