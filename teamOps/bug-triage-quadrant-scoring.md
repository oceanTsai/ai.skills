# Bug 難度量化與四象限派工 Skill

一套將「主觀難度」轉成「可計算數值」的評分方法，用於軟體 bug 或線上問題的自動分類與派工。核心設計是：**五個獨立指標 → 加權合成兩軸分數 → 落入四象限 → 對應工程師水平**。這個 skill 最初是為了一個前端團隊的線上問題自動派工系統設計的（Teams webhook → GAS → OpenAI → Jira），但方法本身適用於任何需要量化 bug 的場景。

## 什麼時候該用這個 skill

- 使用者想把「這個 bug 有多難」變成可以自動化的邏輯
- 使用者手上有既有的人工派工流程，想要換成 AI 或程式判斷
- 使用者要設計 AI system prompt，讓 AI 對 issue 給結構化評分
- 使用者在思考「該讓誰處理」的規則，需要一套水平對應機制
- 使用者對既有評分系統不滿意，想討論調整權重或門檻

## 核心設計哲學

- **主觀切成客觀** — 「難」是主觀的，但「修這個要動幾個檔案」「有沒有穩定重現步驟」是客觀的。指標拆得越細，AI 或人給分越穩定。
- **正交軸分離** — 「技術難度」和「業務急迫度」是兩件事，不能混在一起評分。前者影響「誰能修」，後者影響「什麼時候修」。
- **AI 只評分，程式算象限** — 讓 AI 做它擅長的事（讀 issue 內容給每個維度打分），象限判定用程式跑，這樣門檻/權重可以隨時調整不用重訓 AI。
- **兜底策略要明確** — 派工邏輯永遠會遇到「這個象限沒人」的極端情況，設計時就要規劃 fallback，不能讓系統丟 exception。

## 五個評分維度

每個維度給 1～3 分（低/中/高），這是這套系統的評分基本單位。刻意用三分制而非五分制或十分制，因為 AI 給細粒度分數不穩定，三分制的一致性最高。

### 技術面三指標（決定 X 軸：修復難度）

| 指標 | 說明 | 1 分 | 2 分 | 3 分 |
|---|---|---|---|---|
| `repro` 重現難度 | 能不能穩定讓 bug 出現 | 隨手可重現 | 需特定流程 | 隨機或環境依賴 |
| `fixComplexity` 修復複雜度 | 要動多少程式碼、跨多少模組 | 少量代碼 | 跨多文件 | 涉及核心邏輯或架構 |
| `risk` 修復風險 | 改了以後有沒有機會引發其他問題 | 幾乎無副作用 | 可能影響鄰近模組 | 可能引入連鎖問題 |

### 業務面兩指標（決定 Y 軸：急迫程度）

| 指標 | 說明 | 1 分 | 2 分 | 3 分 |
|---|---|---|---|---|
| `impact` 影響範圍 | 這個 bug 影響多少用戶或系統範圍 | 單模組少數用戶 | 多模組 | 整體系統或多數用戶 |
| `urgency` 業務緊急度 | 業務方的緊迫程度 | 可延後 | 需較快處理 | 須立即修復 |

### 為什麼是這五個維度

技術面刻意選三個，因為它們代表三種不同的難：
- `repro` 是「debug 難度」的代理指標——重現不了就無法修
- `fixComplexity` 是「動手難度」——知道怎麼修但要花很多工
- `risk` 是「決策難度」——技術上簡單但改了怕炸

業務面只給兩個，因為業務判斷本來就比較簡單粗暴——「影響多少人」和「有多急」就夠了，再多指標會產生同義維度。

## 兩軸加權合成

分數不是簡單平均，而是加權：

```
difficulty (X 軸) = fixComplexity × 0.5 + risk × 0.3 + repro × 0.2
priority   (Y 軸) = impact × 0.6 + urgency × 0.4
```

### 為什麼是這個權重

- **fixComplexity 權重最高（0.5）**：實際「要花多少工」是修 bug 最主要的難度來源
- **risk 中等（0.3）**：改壞的風險影響「誰能改」，但不是每個 bug 都有高風險
- **repro 最低（0.2）**：重現困難確實麻煩，但一旦重現，後面就是 fixComplexity 的事
- **impact 高於 urgency（0.6 vs 0.4）**：影響範圍大 = 業務方一定會催；業務方主動說很急但只影響一個人 = 不見得該優先

權重不是絕對的，這是「預設值」。如果團隊處理的 bug 類型有特殊性（例如金融產品 risk 一定要拉到 0.5 以上），要調就直接調。

## 四象限與門檻

兩軸各自 >= `AXIS_THRESHOLD`（預設 2.0）視為「高」：

```
              低急迫          高急迫
高難度    Q2 排程深修      Q1 立刻救火
低難度    Q4 日常練手      Q3 快速處理
```

門檻預設 2.0 = 三分制的中間值，代表「超過平均就算高」。實務上跑一陣子後可調整：

- 太多 Q1/Q2（救火單太多） → 調高門檻到 2.3
- 太少 Q1/Q2（大單看起來都不緊急） → 調低到 1.8

## 象限對應水平

象限決定「誰有資格接這張單」。用職級代號（例如 L1～L5）而不是稱呼（資淺/中階等），程式好比對，稱呼可以另存一欄給人讀。

```javascript
Q1: [L5, L4]           // 高難度高急迫：只有資深以上
Q2: [L5, L4, L3]       // 高難度低急迫：中高階以上可排
Q3: [L3, L2, L1]       // 低難度高急迫：中階以下衝速度
Q4: [L2, L1]           // 低難度低急迫：資淺練手
```

Q1 和 Q2 都要資深以上，差別在於 Q1 要**立刻停下手邊事處理**、Q2 是**排進 Sprint**。Q3 和 Q4 都給中階以下，差別在 Q3 是**當天要解決**、Q4 是**Backlog 慢慢做**。

## AI Prompt 設計原則

給 AI 用的 system prompt 應該包含：

1. **每個指標的三分制定義**（複製上面的表格文字進去）
2. **domain 判斷**（frontend / backend / fullstack / unknown）——影響派工時的第二層篩選
3. **confidence 自評**（1～3）——讓 AI 承認自己資訊不足，程式端可以決定要不要人工覆核
4. **輸出強制 JSON**（`response_format: { type: "json_object" }`），並在 prompt 明確要求「嚴禁輸出 JSON 以外任何內容」
5. **保守原則**：資訊不足時分數傾向 1 或 2 而非 3，同時降低 confidence——避免 AI 在資訊不明時瞎猜給高分導致誤判象限

參考輸出結構：

```json
{
  "summary": "問題摘要，50字內",
  "category": "UI顯示問題 | API資料錯誤 | ...",
  "domain": "frontend | backend | fullstack | unknown",
  "scores": {
    "repro": 1, "fixComplexity": 1, "risk": 1,
    "impact": 1, "urgency": 1
  },
  "confidence": 1,
  "missingInfo": ["缺少的資訊，最多5點"],
  "suggestedQuestions": ["可回覆開單者的問題"],
  "reasoning": "評分理由，30字內"
}
```

## 派工三層篩選

從 AI 給的評分 + 值星表挑人：

1. **象限篩水平** → 用 `QUADRANT_LEVEL[象限]` 過濾
2. **domain 篩領域** → frontend 找 frontend + fullstack；backend 找 backend + fullstack；fullstack/unknown 不限
3. **最少分配數** → 剩下的候選人中挑目前已派單數最少的，平手隨機

### 兜底策略（順序不能亂）

- 兜底 1：domain 篩完沒人 → 只用水平篩（可能是 domain 欄位資料不齊）
- 兜底 2：還是沒人 → 全體都可以接（避免派工失敗丟 exception）

兜底存在的意義：這是自動化系統，不能因為值星表有一格沒填就整個掛掉。

## 常見的設計陷阱

- **讓 AI 直接判斷象限** → 不要。AI 判斷指標分數已經夠了，象限用程式算才能穩定調整規則
- **五分制或十分制** → AI 分數會漂移，三分制的一致性最好
- **只給難度不給急迫** → 只有一個軸的話沒辦法區分「難但不急」和「難又急」，這是完全不同的處理策略
- **權重總和不等於 1** → 不影響象限判定（因為門檻也會跟著變），但分數失去「1～3 分」的直覺意義
- **忘記兜底** → 極端情境（全員都不值星、值星表空、domain 都沒填）會讓系統爆炸
- **cofidence 沒被用** → AI 自評信心度應該反映在通知或介面上，讓值星人員知道這張單需不需要人工覆核

## 適用範圍與限制

- 這套方法適合**線上問題**（有明確 issue 單、可以逐張評分）
- **不適合**：批量的技術債（沒有單張 issue）、需求類的 ticket（不是 bug）、緊急事件（P0 直接電話叫人，不走派工）
- AI 對「修復複雜度」「風險」的評分**先天有限**，因為它看不到程式碼。初期建議把 AI 評分當初判，累積 1～2 個月資料後回頭校準權重
- 如果團隊規模很小（例如 3 人以下），四象限對應水平會變得沒意義，直接用「最少分配」就夠了
